Packaging & deploy

“It works on my machine” is not a deployment strategy. Containers make your service run the same everywhere, and a little discipline with config and secrets keeps it safe. This lesson gets your API off your laptop.

Why containers: kill “works on my machine”

Your service depends on a specific Python version, specific libraries at specific versions, and system bits underneath. Move it to another computer and any mismatch breaks it — the classic “works on my machine” failure. A container solves this by packaging your code together with its entire environment (Python, libraries, config) into one portable image that runs identically on your laptop, a teammate’s machine, or a cloud server. Docker is the standard tool for building and running these images. The payoff: build once, run anywhere, with no “but it worked here” surprises.

You describe the environment in a Dockerfile — a recipe:

Dockerfile
FROM python:3.11-slim              # a known, fixed base
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt   # pinned versions = reproducible
COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

docker build -t my-llm-app . bakes the image; docker run -p 8000:8000 my-llm-app runs it. That image is now a sealed unit you can hand to any host and it just works — which is exactly what deployment platforms expect.

Config and secrets: never bake them in

Your service has settings that change per environment — which model, which database URL, and (if you ever use a paid API) an API key. The rule: configuration comes from the environment, never from hardcoded values in your code. You read settings from environment variables:

config.py
import os

MODEL = os.environ.get("MODEL", "llama3.2")      # default for local
DB_URL = os.environ.get("DB_URL", "sqlite:///local.db")
API_KEY = os.environ.get("API_KEY")          # NEVER a literal in code

This matters for two reasons. First, secrets: a hardcoded API key committed to a repo is a leak — anyone with the code has your credentials, and public GitHub repos get scraped for keys within minutes. Keep secrets in environment variables (or a secret manager), and add files like .env to .gitignore so they’re never committed. Second, portability: the same image runs against your local SQLite in dev and a real database in production, just by changing environment variables — no code change, no rebuild.

Where to deploy (free options)

A container can run on many hosts. For learning, free tiers are plenty: Render, Railway, and Fly.io all deploy a Docker image or a repo to a public URL on a free/starter plan. Push your code, point the platform at it, set your environment variables in their dashboard, and you have a live endpoint. (Heavier local models need more RAM than free tiers give — a good reason your app should read MODEL from config, so it can use a small hosted or local model in dev and swap in production.) You don’t have to deploy anything to complete this course, but knowing the path — container → host → URL → env-var config — is exactly what interviews probe.

The leak that ends interviews

A hardcoded API key (api_key = "sk-abc123...") pushed to GitHub is the single most common, most damaging beginner mistake. Reviewers look for it. Secrets live in environment variables and secret managers — never in code or version control.

Key points
  • Containers (Docker) package code + its whole environment so it runs identically everywhere — no “works on my machine.”
  • A Dockerfile is the recipe; pinned requirements.txt versions make builds reproducible.
  • Config from environment variables, never hardcoded — same image runs in dev and prod by changing env vars.
  • Secrets never go in code or git. Use env vars / secret managers; .gitignore your .env.
  • Deploy a container to a free host (Render / Railway / Fly.io) to get a public URL.
Practice

A teammate’s service has API_KEY = "sk-live-9f3..." at the top of main.py, and the database URL is hardcoded to their laptop’s local Postgres. List what’s wrong and how you’d fix each issue.

▸ Show solution & reasoning

Problem 1 — the hardcoded API key: it’s a secret leak. If this file is in git (especially anything public), the key is compromised and must be rotated immediately. Fix: read it from os.environ["API_KEY"], set the real value as an environment variable on the host, and add any .env file to .gitignore. Consider a secret manager for production.

Problem 2 — the hardcoded DB URL: it ties the code to one machine, so it can’t run anywhere else without editing source. Fix: read DB_URL from the environment with a sensible local default. Now the same image points at local SQLite in dev and the real database in prod purely via config.

Why this is the pattern: code should be identical across environments; only configuration differs, and it flows in from outside. That separation is what makes a service portable, safe to open-source, and deployable to any host — and getting it right is a basic professional expectation that reviewers specifically check.

Why package an LLM service in a Docker container instead of just copying the Python files to a server?

  • A) The container bundles the exact environment (Python, libraries, versions), so it runs identically everywhere — eliminating environment mismatches.
    Correct. Copying just the files leaves the target machine to supply the right Python and library versions, which is where “works on my machine” bugs come from. A container ships the whole environment as one reproducible image, so dev and prod behave the same.
  • B) Containers make the model’s answers more accurate.
    Containers are about reproducible environments and deployment, not model quality. Accuracy comes from your prompts, retrieval, and evals — unchanged by packaging.
  • C) Docker stores your API keys securely for you.
    Docker doesn’t manage secrets for you — in fact, baking secrets into an image is a mistake. Secrets belong in environment variables or a secret manager, injected at run time.
  • D) Containers remove the need for a requirements file.
    The opposite — a pinned requirements.txt is what makes the image reproducible. The Dockerfile installs from it during the build.

Why package an LLM app in a container for deployment?

  • It bundles the app with its exact dependencies so it runs the same everywhere.
    Correct. Containers give reproducible, portable runtimes.
  • It makes the model smarter.
    Packaging is about deployment, not capability.
  • It removes the need for monitoring.
    You still monitor in production.
  • It quantizes the model.
    Unrelated — that's a separate technique.
Deep dive: the twelve-factor habits that make services deployable

What you just did — config from the environment, dependencies pinned, one command to run — are pieces of a widely used checklist for cloud services often called the “twelve-factor app.” A few of its ideas are worth carrying everywhere. Strict separation of config from code (what you practiced) means the same build runs in every environment.

Statelessness means your service shouldn’t store important data in its own memory or local disk between requests, because a deployment might run several copies of it and replace them at any time; anything that must persist goes to a database or object store. This is what lets a platform run three, then ten, copies of your container behind a load balancer to handle traffic — horizontal scaling — without any of them stepping on each other.

For LLM services specifically, statelessness has a twist worth naming in interviews: conversation “memory” must live outside the process (a database or cache keyed by session), never in a local variable, precisely because the next request might hit a different container. Combine that with the operational ring from the last lesson — timeouts, rate limits, caching, streaming — and you have the shape of a real deployment: a stateless, containerized, env-configured service with persistence and state pushed to backing stores. You don’t need to memorize the whole twelve-factor list; you need the instinct that code is fixed, config and state live outside it, because that instinct is what makes something you can actually run at scale.

Your service can run anywhere. But every change you make risks breaking quality — so before anything ships, it should have to pass your evals. Next: CI for prompts & evals.
