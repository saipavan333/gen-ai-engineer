Interview check: LLMOps

“Can you take this prototype to production?” is the question that separates a hobbyist from a hire. This track is your answer. Here’s how teams probe it, with strong responses — and another area where your engineering background is a real edge.

How to use this

Answer out loud before revealing each model answer. The bar for this track: could you take a colleague’s working notebook and describe every step to get it running reliably, safely, and observably in production? That narrative is what wins these rounds.

The questions teams ask

Q1 · The core question

“I have a RAG prototype in a notebook. Walk me through getting it to production.”

▸ Strong answer & what they’re testing

Answer: Wrap the logic in a web service (FastAPI) behind a clean endpoint with validated inputs. Move all config and secrets to environment variables — nothing hardcoded. Package it in a Docker container so it runs identically everywhere, and deploy that to a host. Put a golden-set eval in CI so no change ships without passing a quality gate. Add the operational ring: timeouts, rate limits, caching, and streaming for latency. Finally, instrument it — traces, cost/latency, quality signals — and close the loop by turning production failures into new eval cases. I’d roll changes out to a small slice of traffic first.

Testing: Whether you have the whole lifecycle — service, packaging, secrets, CI eval gate, monitoring — not just “deploy the script.”

Q2 · Packaging

“Why containerize it? Why not just copy the files to a server?”

▸ Strong answer & what they’re testing

Answer: A container bundles the code with its exact environment — Python version, pinned libraries, system deps — so it runs identically on my laptop, CI, and prod, eliminating “works on my machine.” Copying files leaves the target to supply a matching environment, which is where subtle breakages come from. The container is also the unit every deploy platform and autoscaler expects.

Testing: That you understand reproducibility, not just that Docker is fashionable.

Q3 · Secrets (they’re watching for this)

“Where do API keys and database credentials live?”

▸ Strong answer & what they’re testing

Answer: In environment variables or a secret manager, injected at runtime — never hardcoded in source and never committed to version control. Code reads config from the environment; .env files are gitignored. A key in a repo (especially public) is compromised the moment it’s pushed and must be rotated. Same image, different env vars per environment.

Testing: Basic security hygiene. Saying “hardcode it for now” is an instant red flag; “environment variables, never in git” is the expected answer.

Q4 · The differentiator

“How do you stop a prompt change from silently degrading quality?”

▸ Strong answer & what they’re testing

Answer: An eval gate in CI. Prompts live in version control, so a change is a reviewable diff. CI runs deterministic tests plus my golden-set evaluation and fails the build if the score drops below a threshold — so a change that fixes one case but breaks others is blocked before it ships. Every production failure gets added to the golden set, so we can never regress on it again. This is also how I’d safely evaluate a new model: run it against the golden set.

Testing: Whether you know LLM regressions are silent (no error) and that the answer is scored evals gating deploys — a senior signal most candidates miss.

Q5 · Monitoring

“It’s live. What do you monitor, and what’s the difference between offline and online evals?”

▸ Strong answer & what they’re testing

Answer: Traces of every request (prompt, context, output, tokens, timing), cost and latency per call, quality signals (user feedback, online-eval scores on sampled traffic), and error/retry/refusal rates. Offline evals run on a fixed golden set in CI and answer “did this change regress known cases?” — they gate deploys. Online evals run on live traffic and answer “how are we doing right now on real inputs?” — they catch drift and novel failures the golden set never had. They feed each other: online failures become offline test cases.

Testing: That you distinguish regression-gating (offline) from reality-checking (online), and know traces are how you debug specific bad answers.

Q6 · Judgment

“Your LLM feature is slow and expensive in production. Walk me through fixing it without guessing.”

▸ Strong answer & what they’re testing

Answer: Measure first — use traces to find where the tokens and time actually go (long prompts? many agent steps? a slow model?). Then apply the targeted lever from Track 8: cache repeated calls, route easy requests to a smaller model, trim the prompt/context, shorten outputs, and stream for perceived latency. Re-measure to confirm, and make sure the eval score didn’t drop as a result — a cheaper setup that’s worse isn’t a win. Optimization is measured, not guessed.

Testing: That you connect monitoring (find the bottleneck) to serving levers (Track 8) and re-verify quality — engineering discipline, not vibes.

Rapid-fire self-check

What’s the mechanism that lets an “eval gate” actually block a bad change in CI?

  • A) The eval script exits with a non-zero (failure) code when the score is below the threshold, which fails the CI job and blocks the deploy.
    Correct. CI treats a non-zero exit as failure. Your eval runs the golden set and exits 1 if the score is too low — turning a quality drop into a red build, exactly like a failing test.
  • B) CI reads the model’s mind to detect bad prompts.
    There’s nothing magic — the gate is a concrete script that scores a golden set and signals pass/fail via its exit code. Quality has to be measured, then enforced.
  • C) It emails users to ask if the change is good.
    The whole point is to catch regressions before users see them. The gate is automated and runs at build time, not a user survey.
  • D) It only works if outputs are byte-for-byte identical every run.
    Evals handle non-determinism by scoring quality (exact match where appropriate, or an LLM judge) and gating on an aggregate threshold — they don’t require identical outputs.

Where should a chatbot’s conversation history be stored in a scaled, containerized deployment?

  • A) In an external store (database/cache) keyed by session — not in the process’s memory.
    Correct. A scaled service runs multiple interchangeable container copies, and the next request may hit a different one. State kept in a local variable would be lost or inconsistent, so conversation memory must live outside the process (stateless service, external state).
  • B) In a global variable inside the service, for speed.
    That breaks as soon as there’s more than one container (or the container restarts): different copies wouldn’t share the variable. Local in-memory state is the classic scaling bug.
  • C) In the Docker image itself.
    The image is a read-only build artifact, identical for every copy — it can’t hold per-user, changing conversation state. Runtime state belongs in an external store.
  • D) It shouldn’t be stored at all.
    Multi-turn chat needs history to work; you just store it outside the process (keyed by session) so any container copy can serve the next turn.

Your mean latency looks fine on the dashboard, but users complain about slowness. What do you check?

  • The tail — p95 and p99 — because the mean hides a slow minority.
    Correct. Most real incidents are a slow tail, not a uniform slowdown; alert on percentiles, not the average.
  • The mean again, more carefully.
    The mean is precisely what’s hiding the problem.
  • The total token count.
    Doesn’t reveal the latency distribution.
  • The model’s temperature.
    Unrelated to latency tails.

Your offline evals still pass in CI, but production answer quality has clearly dropped. Most likely cause?

  • Drift — the golden set is a frozen snapshot; you need online evals on live traffic to catch new failure modes.
    Correct. Offline evals prove you didn’t regress on known cases; only monitoring live traffic reveals the unknown ones (new question types, stale data).
  • The CI server is broken.
    Passing offline evals with degraded production is the classic drift signature, not CI failure.
  • The model got smaller.
    Nothing here implies a model change.
  • The golden set is too large.
    Size isn’t the issue; coverage of new cases is.
Can you explain each of these? (the real test)
  • Notebook → FastAPI service → Docker → deployed URL.
  • Why containers give reproducibility; config & secrets from the environment, never in git.
  • The eval gate in CI (non-zero exit blocks deploy) and why LLM regressions are silent.
  • Monitoring: traces, cost/latency, quality signals; offline vs online evals.
  • The feedback loop: production failures become golden-set cases.
  • Stateless services & external state for chat memory at scale.
