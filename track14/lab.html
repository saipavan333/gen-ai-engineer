Lab: Build a production monitor

Averages lie. You’ll generate a realistic request log, compute the metrics that actually matter (p50, p95, error rate, cost/day), write alert rules — then simulate a bad day and watch your monitor catch a regression no average would have shown.

How to use this lab

Run the cells in order — they share memory. Plain Python, running here.

Step 1 — Generate a request log

Every production LLM call should emit a trace: tokens in, tokens out, latency, success. Here’s a day of them.

step1_log.py
import random
random.seed(7)

def make_log(n=500, bad_day=False):
    rows = []
    for _ in range(n):
        in_tok  = random.randint(400, 1200)
        out_tok = random.randint(80, 300)
        latency = 0.25 + in_tok / 1000 * 0.12 + out_tok * 0.012
        if bad_day and random.random() < 0.15:
            latency *= 3                                  # a slow tail appears
        error = random.random() < (0.06 if bad_day else 0.01)
        rows.append({"latency": round(latency, 3), "in": in_tok,
                     "out": out_tok, "error": error})
    return rows

monday = make_log()
print("requests:", len(monday))
print("sample  :", monday[0])

Step 2 — The metrics that matter

Report percentiles, not means. The average hides the users having a terrible time; p95 is where they live.

step2_metrics.py
def pct(values, p):
    v = sorted(values)
    k = int(round((p / 100) * (len(v) - 1)))
    return v[k]

def summarize(log):
    lat    = [r["latency"] for r in log if not r["error"]]
    errors = sum(1 for r in log if r["error"])
    cost   = sum((r["in"] * 0.15 + r["out"] * 0.60) / 1000000 for r in log)
    return {"mean": round(sum(lat) / len(lat), 2),
            "p50": round(pct(lat, 50), 2),
            "p95": round(pct(lat, 95), 2),
            "error_rate": round(errors / len(log), 3),
            "cost_per_day": round(cost, 4)}

mon = summarize(monday)
print(mon)

Step 3 — Alert rules

A dashboard nobody watches is decoration. Thresholds turn metrics into something that wakes you up.

step3_alerts.py
RULES = {"p95": 3.0, "error_rate": 0.03, "cost_per_day": 0.20}

def check_alerts(metrics):
    fired = []
    for key, limit in RULES.items():
        if metrics[key] > limit:
            fired.append(key + " = " + str(metrics[key]) + " exceeds " + str(limit))
    return fired

print("monday alerts:", check_alerts(mon) or "none - healthy")

Step 4 — A bad day arrives

Now a day where a slow tail and more errors creep in. Watch which number moves first — and notice how little the mean reacts.

step4_regression.py
tuesday = make_log(bad_day=True)
tue = summarize(tuesday)
print("tuesday:", tue)
print("alerts :", check_alerts(tue) or "none")

print("\nchange vs monday")
for key in mon:
    print("  ", key, mon[key], "->", tue[key],
          "  delta", round(tue[key] - mon[key], 4))
The lesson in the numbers

The mean barely twitched while p95 jumped — the slow tail hit a minority of users hard, which is exactly the shape of most real incidents. Alert on percentiles and error rate; use the mean only for capacity planning.
