Track 10 · Capstone 10.2

Capstone: a tool-using agent

Build an agent that actually does things: a guarded ReAct loop that calls real tools (a calculator and a lookup), reasons over the results, and refuses to run away or do anything dangerous. It demonstrates Track 4 end to end and shows you can give a model autonomy responsibly — runs free on local models.

What you’re building

An agent that answers questions requiring computation or fresh lookups by choosing and calling tools in a loop — with the production guards that separate a toy from something shippable: a step limit, argument validation, and human approval for risky actions (Track 4.3, 4.8). Free, local, via Ollama’s tool calling.

Setup (free)
$ pip install ollama wikipedia
$ ollama pull llama3.2          # supports tool calling, runs locally

Step 1 — Define tools (and mark the risky one)

tools.py
import wikipedia

def calculator(expression: str) -> str:
    # safe-ish eval: digits and operators only (validate args before running!)
    if not set(expression) <= set("0123456789+-*/(). "):
        return "error: invalid characters"
    return str(eval(expression))

def wiki_lookup(query: str) -> str:
    try:
        return wikipedia.summary(query, sentences=2)
    except Exception as e:
        return f"error: {e}"

def send_email(to: str, body: str) -> str:        # RISKY: irreversible, needs approval
    return f"(pretend) email sent to {to}"

TOOLS = {"calculator": calculator, "wiki_lookup": wiki_lookup, "send_email": send_email}
RISKY = {"send_email"}                              # gate these behind a human

SCHEMAS = [
    {"type": "function", "function": {"name": "calculator",
        "description": "Evaluate an arithmetic expression.",
        "parameters": {"type": "object", "properties": {"expression": {"type": "string"}}, "required": ["expression"]}}},
    {"type": "function", "function": {"name": "wiki_lookup",
        "description": "Look up a fact on Wikipedia.",
        "parameters": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}}},
]   # note: we DON'T advertise send_email unless the task needs it (least privilege)

Step 2 — The guarded agent loop

agent.py
import ollama
from tools import TOOLS, RISKY, SCHEMAS

def run_agent(goal, max_steps=6):
    msgs = [{"role": "system", "content": "Use tools to solve the task. Think, then act. Answer when sure."},
            {"role": "user", "content": goal}]
    for step in range(max_steps):                        # GUARD: bounded loop
        m = ollama.chat(model="llama3.2", messages=msgs, tools=SCHEMAS)["message"]
        msgs.append(m)
        calls = m.get("tool_calls")
        if not calls:
            return m["content"]                              # final answer
        for call in calls:
            name = call["function"]["name"]
            args = call["function"]["arguments"]
            if name not in TOOLS:                           # GUARD: allowlist
                result = f"error: unknown tool {name}"
            elif name in RISKY and input(f"Approve {name}({args})? [y/N] ") != "y":
                result = "denied by human"               # GUARD: human-in-the-loop
            else:
                print(f"  ↳ {name}({args})")               # trace every action (debuggability)
                result = TOOLS[name](**args)
            msgs.append({"role": "tool", "content": str(result)})  # observation back in
    return "Stopped: hit max steps."                      # GUARD: never loop forever

if __name__ == "__main__":
    print(run_agent("What is the population of France, multiplied by 2?"))
    # → wiki_lookup('France population') → calculator('... * 2') → final answer
What makes this portfolio-grade
  • Every guard is present and labeled: max-steps, allowlist, argument validation, human approval, full tracing.
  • Least privilege: the risky send_email tool isn’t even advertised unless needed.
  • The trace prints each action — you can show the reason → act → observe loop live.
  • It runs free and local, and you can narrate exactly how injection or runaway loops are contained.
Extensions

1. Add an eval (Track 6): a set of tasks with known answers; measure success rate, average steps, and whether it called valid tools — run it on every change.
2. Add a cost/step budget (Track 8) that stops the agent if it exceeds a token cap.
3. Add memory (Track 4.5): summarize the trace when it grows so long tasks don’t blow the context.
4. Treat tool results as untrusted (Track 7): what if a Wikipedia page contained “ignore your instructions”? Confirm your guards contain it.

▸ The interview narrative

“I built a tool-using agent as a guarded ReAct loop. The model only proposes actions; my code disposes — I allowlist tools, validate arguments, cap steps, and require human approval for anything irreversible like sending email, which I don’t even expose unless the task needs it (least privilege). I log every action so I can read the trace when it misbehaves. Because injection can’t be fully prevented, I designed so a hijacked model can’t do harm — that containment is the whole point.” This single project lets you speak to tool calling, the loop, planning, memory, MCP, failure modes, and agent security (all of Track 4) from experience.

In this agent, why is send_email deliberately left out of the advertised SCHEMAS for general tasks?

  • A) Least privilege — don’t expose a risky, irreversible tool unless the task needs it, so a hijacked or confused model can’t reach it.
    Correct. The fewer dangerous capabilities the model can access, the smaller the blast radius of any injection or error. Not advertising the risky tool (and gating it behind approval when it is used) is exactly the least-privilege containment that makes agents safe.
  • B) Because the model can’t understand email tools.
    It could use the tool fine — the reason it’s withheld is safety (least privilege), not capability.
  • C) To save tokens in the prompt.
    Schema size is negligible; the real reason is limiting access to a dangerous, irreversible action. Security, not token cost.
  • D) Because email tools are slow.
    Speed isn’t the concern. The point is containing what a manipulated model can do by not exposing risky capabilities.
