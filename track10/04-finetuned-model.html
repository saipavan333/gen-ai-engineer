Track 14 · Capstone 14.4

Capstone: a fine-tuned domain model

Fine-tune a small open model to adopt a specific behavior — a consistent format, tone, or narrow skill — using QLoRA on a free Colab GPU, then prove it improved without breaking general ability. This demonstrates Track 5 end to end and shows you can do the thing most candidates only talk about.

What you’re building

A LoRA/QLoRA fine-tune of a small model (e.g., Qwen2.5-0.5B) on a few hundred instruction→response examples that teach a behavior — say, replying in a strict support format. Then an evaluation comparing the fine-tuned model to the base on a held-out set, checking the target task improved and general ability didn’t regress. All free on Colab’s GPU.

Step 2 — QLoRA fine-tune (free GPU)

train.py
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft import LoraConfig
from trl import SFTTrainer, SFTConfig

model_id = "Qwen/Qwen2.5-0.5B"
bnb = BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type="nf4",
                         bnb_4bit_compute_dtype=torch.bfloat16)       # QLoRA
model = AutoModelForCausalLM.from_pretrained(model_id, quantization_config=bnb, device_map="auto")
tok = AutoTokenizer.from_pretrained(model_id)

trainer = SFTTrainer(model=model, train_dataset=train,
    peft_config=LoraConfig(r=16, lora_alpha=32, task_type="CAUSAL_LM"),
    args=SFTConfig(output_dir="adapter", num_train_epochs=3, learning_rate=2e-4))
trainer.train()
trainer.save_model("adapter")                # a few MB — the whole "fine-tune"

What makes this stand out
  • You actually fine-tuned a model — most candidates have only read about it.
  • You used QLoRA on a free GPU — demonstrating you know the modern, accessible method.
  • You evaluated against the base on held-out data AND checked for regression — the rigorous, honest part.
  • You can articulate why you fine-tuned (behavior, not knowledge) vs. when you’d use RAG instead (Track 5.1).
  • The adapter is a tiny artifact you can show and explain.
Extensions

1. Replace the format check with an LLM-as-judge (reuse your harness from 14.3) for richer quality scoring.
2. Demonstrate prompt-shrinking (Track 8.7): show the fine-tuned model needs a shorter prompt than the base to get the same behavior — quantify the token savings.
3. Try distillation (Track 5.1 deep dive): use a bigger model to generate training data for the small one.
4. Combine with RAG (14.1): fine-tune for format/tone, retrieve for facts — the production pattern.

Your fine-tune improves the target format from 0.40 → 0.92. Before you call it a success, what must you also check?

  • A) That general ability didn’t regress — score both models on diverse off-task prompts to rule out catastrophic forgetting.
    Correct. A big target-task gain can hide a loss of general ability from over-specialization. Until you’ve confirmed the fine-tune didn’t get worse at off-task behavior, you don’t know it’s a net win — always measure both the target task and general ability against the base.
  • B) Nothing — a 0.40 → 0.92 jump is conclusive.
    A target jump alone can mask a regression elsewhere. Shipping on one metric is how teams deploy a model that’s great at one thing and worse at everything else.
  • C) That the adapter file is small.
    Adapter size is an efficiency detail, irrelevant to whether the model is better overall. The real check is general-ability regression.
  • D) That training loss reached zero.
    Near-zero training loss is a warning sign (memorization), not a success criterion. Held-out target quality and general ability are what matter.
