Lab: Build an agent loop with real tools

An agent is a loop: think, act, observe, repeat. You’ll write real tools, a planner that decides which to call, the loop that runs them — and then the guards that stop it spinning forever, which is what separates a demo from something you’d let run unattended.

How to use this lab

Run the cells in order — they share memory. The planner is a small rule-based stand-in for a model, so the loop mechanics stay visible.

Step 1 — Give it tools

A tool is just a function plus a contract. Note the calculator refuses anything but arithmetic characters — a tool should never be a hole through which arbitrary code escapes.

step1_tools.py
def tool_calculator(expr):
    allowed = set("0123456789+-*/(). ")
    if not set(expr) <= allowed:          # refuse anything but arithmetic
        return "error: unsupported characters"
    try:
        return str(eval(expr, {"__builtins__": {}}, {}))
    except Exception as e:
        return "error: " + str(e)

ORDERS = {"A100": "shipped", "B200": "processing"}

def tool_order_status(order_id):
    return ORDERS.get(order_id.strip().upper(), "unknown order")

TOOLS = {"calculator": tool_calculator, "order_status": tool_order_status}

print(TOOLS["calculator"]("120 * 3"))
print(TOOLS["calculator"]("__import__('os')"))
print(TOOLS["order_status"]("a100"))

Step 2 — A planner that decides

The planner looks at the question and what it has already observed, then returns either an action or a final answer. A real model returns this as text you parse; the decision shape is identical.

step2_planner.py
def planner(question, observations):
    q = question.lower()
    used = [t for t, _ in observations]
    if "order" in q and "order_status" not in used:
        order_id = q.split()[-1].strip("?.")
        return ("action", "order_status", order_id)
    if "total" in q and "calculator" not in used:
        return ("action", "calculator", "3 * 120")
    if observations:
        return ("final", None, "Based on the tool result: " + observations[-1][1])
    return ("final", None, "I don't know")

print(planner("What is the status of order A100?", []))
print(planner("What is the status of order A100?", [("order_status", "shipped")]))

Step 3 — Run the loop

Act, observe, feed the observation back, decide again. Notice the max_steps cap: an agent without one is an outage waiting to happen.

step3_loop.py
def run_agent(question, max_steps=4):
    print("Q:", question)
    observations = []
    for step in range(max_steps):
        kind, tool, arg = planner(question, observations)
        if kind == "final":
            print("   FINAL:", arg)
            return arg
        if tool not in TOOLS:
            print("   invalid tool:", tool)
            return "error"
        result = TOOLS[tool](arg)
        print("   step", step + 1, "| action:", tool, arg, "-> observation:", result)
        observations.append((tool, result))
    print("   step cap reached — escalating to a human")
    return "escalated"

run_agent("What is the status of order A100?")
print()
run_agent("What is the total for 3 items?")

Step 4 — Make it fail, then contain it

Now a broken planner that never finishes — the classic runaway agent. Two guards stop it: a step cap and repeat-detection.

step4_guard.py
def bad_planner(question, observations):
    return ("action", "order_status", "A100")      # never returns 'final'

def run_bounded(question, planner_fn, max_steps=5):
    observations, seen = [], set()
    for step in range(max_steps):
        kind, tool, arg = planner_fn(question, observations)
        if kind == "final":
            return "final: " + arg
        if (tool, arg) in seen:
            return "stopped: repeated the same action, no progress"
        seen.add((tool, arg))
        result = TOOLS[tool](arg)
        print("   step", step + 1, "->", tool, result)
        observations.append((tool, result))
    return "stopped: step cap"

print(run_bounded("status of order A100?", bad_planner))
What you built
  • Tools with contracts and input validation.
  • A planner that chooses an action or finishes.
  • The ReAct loop — act, observe, decide again.
  • Guards: step caps and repeat-detection, so failure is bounded.

Every production agent is this, plus better planning and more tools. The guards are not optional extras — they are the difference between an agent and an incident.

▸ Show reasoning

1. Refunding moves real money and is hard to reverse, so it needs human approval (or a hard limit plus an audit log) before execution. Read-only tools can run freely; write tools should be gated by how costly and irreversible the action is — that risk-tiering is the heart of safe agent design.

2. It runs to the step cap, repeating an identical call each time — every step a real API call you pay for. Without the cap it would loop forever. That's how a demo agent becomes a surprise invoice.
