The agent loop, and how it runs away

An agent is a loop with a budget. Watch a healthy one reason, call tools and stop — then run the same loop with a failing tool and see exactly where the money goes.

How to use this lab

Play the healthy loop first. Then switch to the runaway loop and watch the same agent, with the same prompt, behave completely differently because one tool started failing.

Make it yours

1. In the healthy run, which facts in the final answer came from the model, and which came from an observation?
2. In the runaway run, the agent is not broken — it is doing exactly what it was designed to do. What is the actual bug, and where does it live?

▸ Show reasoning

1. Every fact — the delivery date and the 30-day window — came from an observation. The model supplied only the reasoning that connected them. That is the property you want: a grounded agent should be unable to state a fact it did not fetch.

2. The bug is not in the model. It is the missing guard in your code: no step budget, no repeat detection, no handling for a tool that fails the same way twice. The model will keep trying forever because trying again is a locally sensible move. Bounded loops are not a safety extra, they are the thing that makes an agent shippable.

What you just saw
  • Thought → action → observation, the cycle underneath every agent framework.
  • The model requesting a tool rather than running one — your code stays in control.
  • Every fact in the answer traceable to an observation.
  • A runaway loop, and the budget that stops it.
