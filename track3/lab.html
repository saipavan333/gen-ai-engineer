Lab: Build a tiny RAG pipeline

Chunk, embed, retrieve, ground — the whole pipeline, written by you in plain Python, running in this page. No vector database, no API key. Once you’ve built it small, the production version is just the same shape with better parts.

How to use this lab

Run the cells in order — they share memory like a notebook. Then break things: change the question, drop a chunk, and watch retrieval respond.

Step 1 — Chunk the documents

Retrieval works over chunks, not whole documents, so the model gets a small, relevant slice instead of everything. Here each line is one chunk; real systems split on paragraphs or a fixed size with overlap.

step1_chunk.py
DOC = """Refunds are issued within 5 business days to your original payment method.
You can return any item within 30 days of delivery for a full refund.
Standard shipping is free on orders over $50; express shipping costs $9.99.
Our support team is available Monday to Friday, 9am to 6pm EST.
To reset your password, click Forgot Password on the login page.
Gift cards are non-refundable and never expire."""

chunks = [c.strip() for c in DOC.split("\n") if c.strip()]
for i, c in enumerate(chunks):
    print(i, "|", c[:58])
print("total chunks:", len(chunks))

Step 2 — Turn text into vectors, and compare them

Real systems use an embedding model; the idea is identical and you can build it here: represent each chunk as a bag of meaningful words, then measure similarity as the cosine of the angle between two vectors. Same maths, smaller brain.

step2_vectors.py
import math, re
from collections import Counter

STOP = {"the","a","an","is","are","to","of","for","on","in","do","my","you","your","can","how","and","it"}

def vec(text):
    words = [w for w in re.findall(r"[a-z0-9]+", text.lower())
             if w not in STOP and len(w) > 2]
    return Counter(words)

def cosine(a, b):
    shared = set(a) & set(b)
    dot = sum(a[w] * b[w] for w in shared)
    na = math.sqrt(sum(v*v for v in a.values()))
    nb = math.sqrt(sum(v*v for v in b.values()))
    return dot / (na * nb) if na and nb else 0.0

index = [(c, vec(c)) for c in chunks]
print("indexed", len(index), "chunks")
print("vector for chunk 0:", dict(vec(chunks[0])))

Step 3 — Retrieve the top-k

Score every chunk against the question and keep the best few. Notice the third question: nothing scores above zero, which is the signal a good system uses to say “I don’t know” instead of inventing an answer.

step3_retrieve.py
def retrieve(query, k=2):
    qv = vec(query)
    scored = [(cosine(qv, cv), c) for c, cv in index]
    scored.sort(reverse=True, key=lambda x: x[0])
    return [(round(s, 3), c) for s, c in scored[:k] if s > 0]

for q in ["How long do refunds take?", "Is shipping free?", "What is the capital of France?"]:
    print("\nQ:", q)
    hits = retrieve(q)
    if not hits:
        print("   (nothing relevant — the system should refuse)")
    for s, c in hits:
        print("  ", s, "|", c[:58])

Step 4 — Ground the answer

The final move: put only the retrieved chunks into the prompt, demand citations, and instruct the model to refuse when the context doesn’t contain the answer. This prompt is what actually stops hallucination.

step4_ground.py
def build_prompt(query, k=2):
    hits = retrieve(query, k)
    if not hits:
        return None
    context = "\n".join(f"[{i+1}] {c}" for i, (s, c) in enumerate(hits))
    return ("Answer using ONLY the context. Cite sources like [1].\n"
            "If the answer is not in the context, say you don't know.\n\n"
            "Context:\n" + context + "\n\nQuestion: " + query + "\nAnswer:")

print(build_prompt("How long do refunds take?"))
print("\n--- an unanswerable question ---")
print(build_prompt("What is the capital of France?"))
What just happened

The last line printed None — no context, so no prompt, so no answer. That refusal path is not a failure; it’s the single most valuable behaviour a grounded system has. A RAG app that guesses when it has nothing is worse than one that admits it.

Make it yours

1. Ask “How do I change my password?” — it retrieves the reset chunk even though the words differ a little. Why does it still work, and where would this simple approach fail?
2. Set k=4 in Step 4. Does the prompt get better or just longer?
3. Add your own document lines to DOC, re-run Steps 1–4, and query them.

▸ Show reasoning

1. It works because “password” overlaps literally. It fails the moment wording diverges — ask “I’m locked out of my account” and this bag-of-words scorer finds nothing, because it matches words, not meaning. That gap is precisely why production RAG uses embedding models: they place “locked out” near “reset password” in vector space. You just felt the limitation that justifies the whole embedding stack.

2. Longer, and usually slightly worse — you drag in irrelevant chunks that distract the model and cost tokens. More context is not more accuracy; that’s why reranking and tight top-k matter (Track 11).

What you built
  • Chunking — splitting documents into retrievable units.
  • Vectors + cosine similarity — the maths behind every embedding search.
  • Top-k retrieval with a real relevance threshold.
  • A grounding prompt with citations and an explicit refusal path.

Swap your vec() for an embedding model and your list for a vector database, and this is production RAG. The shape doesn’t change.
