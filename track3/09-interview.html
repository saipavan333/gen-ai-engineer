Interview check: RAG & Vector Databases

RAG is the most common system-design topic in applied-GenAI interviews, because it touches data, retrieval, prompting, and evaluation all at once. Here are the questions teams ask, the answers that signal seniority, and the debugging instincts they’re really probing for.

The drill

Answer each out loud before opening the model answer. The throughline interviewers listen for: you treat RAG as a retrieval + grounding + evaluation system, you can separate retrieval failures from generation failures, and you make decisions by measurement.

The questions teams ask

Q1 · Framing

“What is RAG, and when would you choose it over a long context window or fine-tuning?”

▸ Strong answer

RAG retrieves relevant text from a knowledge base and puts it in the prompt so the model answers grounded in it. Choose it when knowledge is large, changing, private, or must be cited. Long context suits a few small docs that fit the window; fine-tuning suits durable behavior/format, not fast-changing facts. The mantra: RAG for knowledge, fine-tuning for behavior. RAG also updates by re-indexing — no retraining.

Testing: a crisp decision rule among the three, not “RAG is always best.”

Q2 · End to end

“Walk me through a production RAG pipeline.”

▸ Strong answer

Offline: ingest documents, chunk them (with overlap + metadata), embed each chunk, store vectors in a vector DB. Online, per query: embed the query (same model), retrieve top candidates (often hybrid dense + keyword), rerank the shortlist with a cross-encoder, assemble a grounded prompt with numbered sources, generate with “answer only from context + cite,” and return the answer with citations. Guard each stage with metrics: recall@k/MRR for retrieval, faithfulness/correctness for generation.

Testing: that you know the full pipeline and which metric protects each stage.

Q3 · Chunking

“How do you decide chunk size and overlap?”

▸ Strong answer

Aim for one coherent idea per chunk (often ~200–500 tokens), with overlap so ideas spanning boundaries survive. Prefer recursive/structural splitting (respect paragraphs, headings, code) over blind fixed cuts, and attach metadata (source, id, section) for citations and filtering. Then tune by evaluation: chunk size is a recall@k knob, not a guess. Ensure chunks fit the embedding model’s max length.

Testing: trade-off awareness + “tune with metrics,” not a magic number.

Q4 · Embeddings

“What’s the most common embedding bug, and how do you pick a model?”

▸ Strong answer

The classic bug: embedding documents and queries with different models (or changing models without re-indexing) — the vectors become incomparable and retrieval silently breaks. Use the same model both sides, normalize so dot = cosine, and re-embed everything on a model change. Pick a model on quality vs speed/size, max sequence length (chunks must fit), domain, and language — and validate with recall@k on your own data, not just a leaderboard.

Testing: the same-model rule (a real production footgun) + empirical model choice.

Q5 · Vector DB & security

“How does ANN work, and how do you stop one user retrieving another’s documents?”

▸ Strong answer

ANN indexes (HNSW graph, IVF clusters) find near-nearest neighbors while checking a fraction of vectors — trading a little recall for big speed, tuned via ef/nprobe. For isolation, attach ownership metadata (tenant/user id) and filter every query by it — or use per-tenant indexes for hard isolation. Vector similarity has no notion of permissions, so a missing filter is a data-leak, not just a quality issue.

Testing: recall/latency trade-off + treating multi-tenancy as a security control.

Q6 · Reranking

“Why add a reranker, and when does it not help?”

▸ Strong answer

A bi-encoder embeds query and doc separately (fast, blunt); a cross-encoder reads them together (accurate, slow). Two-stage retrieval fetches many candidates cheaply, then reranks the shortlist for precision — often a bigger quality win than a larger LLM. It only helps if recall is already good: reranking reorders the candidate set but can’t surface a chunk that retrieval never fetched. If recall@k is low, fix retrieval first.

Testing: the bi/cross distinction + the recall-vs-ordering insight.

Q7 · Hybrid search

“When is semantic search not enough?”

▸ Strong answer

When queries contain exact strings — error codes, SKUs, names, rare jargon — embeddings blur them and miss. Keyword search (BM25) nails exact terms but misses synonyms. Hybrid runs both and fuses (RRF combines by rank, avoiding incomparable scores), covering identifier queries and natural-language queries together. Especially valuable for technical/enterprise search where users mix both.

Testing: knowing dense and sparse fail in opposite cases, and why RRF.

Q8 · Grounding

“How do you make answers trustworthy and citable?”

▸ Strong answer

Number the retrieved chunks, instruct the model to answer only from them, cite [n] per claim, and say “not in the documents” when absent. Then verify citations programmatically — that the cited chunk exists and actually supports the claim (overlap or LLM-as-judge faithfulness). Surface sources to users for high-stakes use. The goal isn’t a trustworthy model; it’s a system where every claim is checkable.

Testing: the recipe + verification + the “checkable, not trusted” mindset.

Q9 · The debugging question

“A RAG bot gives a wrong answer. How do you debug it?”

▸ Strong answer

Split the failure. First check retrieval for that query: was the relevant chunk in the top-k (recall)? If not, it’s a retrieval problem — look at chunking, embedding model, hybrid search, or fetch more candidates and rerank. If the right chunk was retrieved, it’s a generation problem — check grounding/faithfulness: is the prompt forcing only-from-context, did the model ignore or misread the chunk, is it citing correctly? Reproduce with the exact retrieved context. Never guess — the recall@k number for that query tells you which half to fix.

Testing: the single most important RAG skill — isolating retrieval vs generation with a measurement.

Rapid-fire self-check

Recall@10 is high, faithfulness is low. The fix is mostly in…

  • A) Generation — grounding and prompting (the right chunks are already retrieved).
    Correct. High recall means evidence reaches the prompt; low faithfulness means the model isn’t using it faithfully. Fix grounding (only-from-context, abstain, verify citations), not retrieval.
  • B) Retrieval — switch embedding models.
    Retrieval is healthy (high recall). Changing the embedder targets a non-problem and ignores the low faithfulness number.
  • C) The vector index — rebuild it.
    Nothing indicates an index issue; recall is high. The signal points squarely at generation faithfulness.
  • D) Chunking — make chunks smaller.
    Chunking affects retrieval, which is already working. Smaller chunks won’t make the model use the supplied context faithfully.

Which combination is the strongest, most cost-aware retrieval pipeline?

  • A) Hybrid (dense + keyword) retrieve a wide shortlist → cross-encoder rerank → top few into a grounded prompt.
    Correct. Hybrid maximizes recall across semantic and exact-match queries, the reranker fixes ordering cheaply (only on the shortlist), and grounding makes the answer faithful and citable. This is the standard production-grade design.
  • B) Run a cross-encoder over the entire corpus for every query.
    Far too slow — that’s a forward pass per document per query. The point of two-stage retrieval is to rerank only a shortlist, not the whole corpus.
  • C) Dense retrieval only, then a much bigger LLM.
    A bigger LLM can’t use chunks that retrieval missed (dense alone misses exact terms), and it’s expensive. Improving retrieval beats upgrading the model for most RAG quality problems.
  • D) Keyword search only, with all results pasted into the prompt.
    Keyword-only misses synonyms/paraphrases, and dumping everything invites “lost in the middle” and high cost. You want hybrid retrieval, reranking, and a focused grounded prompt.
Can you explain each of these?
  • The full RAG pipeline (index once; embed → retrieve → rerank → ground → generate).
  • Chunking trade-offs and why overlap + metadata matter.
  • Same-model embedding rule; choosing a model empirically.
  • ANN recall/latency trade-off and metadata filtering as security.
  • Bi- vs cross-encoder, and reranking reorders but can’t fix recall.
  • Dense vs sparse vs hybrid, and RRF.
  • Grounding recipe, abstention, citation verification, faithfulness.
  • Evaluating retrieval vs generation separately, and the measurement loop.
