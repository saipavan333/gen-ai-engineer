Track 9 · Lesson 9.1

Foundations Q&A

Fundamentals are table stakes: every candidate is expected to know tokens, embeddings, context, and sampling cold. This is a rapid-fire bank to make those answers automatic. Say each one aloud before opening it — fluency, not recognition, is the goal.

How interviewers grade fundamentals

They’re checking that you understand the machine, not that you memorized definitions. Lead with the mechanism (next-token prediction), then the practical consequence (cost, limits, failure modes). A crisp, consequence-aware answer beats a textbook one every time.

The bank

Q · core

“In one breath, how does an LLM generate text?”

▸ Answer

It predicts the next token given the tokens so far — a probability over its whole vocabulary — then samples one, appends it, and repeats (autoregression). All its “knowledge” is patterns learned by doing this over huge amounts of text.

Q · tokens

“What’s a token, and why care about counts?”

▸ Answer

A chunk of text (often a word-piece) from the model’s fixed vocabulary, produced by a sub-word tokenizer (e.g., BPE). Tokens are the unit of cost (billing), context limits (windows), and latency. ~4 chars/token in English; code and many languages cost more. They also explain why models miscount letters — they see tokens, not characters.

Q · embeddings

“What’s an embedding and how do you compare two?”

▸ Answer

A vector representing a text’s meaning, from an embedding model trained so similar meanings get similar directions. Compare with cosine similarity (+1 similar, 0 unrelated, −1 opposite). They power semantic search and RAG — matching meaning even without shared words. (Distinct from the model’s internal token representations; for retrieval we mean text/sentence embeddings.)

Q · context

“What is the context window, and what happens when you exceed it?”

▸ Answer

The max tokens the model can process at once, shared by instructions, history, retrieved context, and the output. Exceed it and the oldest tokens get truncated (chat “memory” is just re-sent history). Bigger isn’t free — more cost, more latency, and “lost in the middle,” which motivates RAG over dumping everything in.

Q · attention

“What is attention, and why is long context expensive?”

▸ Answer

Attention lets each token weigh how much every other token matters, which is how references and meaning get resolved (the transformer’s core). Because every token compares against every other, the work scales with the square of sequence length — doubling context roughly quadruples attention cost. That’s the deep reason long prompts are slow and pricey.

Q · sampling

“Explain temperature and top-p; settings for extraction vs brainstorming?”

▸ Answer

Temperature reshapes the probability distribution before sampling — low sharpens toward the top token (deterministic), high flattens it (varied). Top-p (nucleus) samples from the smallest set of tokens summing to p, adapting to confidence. Extraction/JSON/classification → temperature 0 (consistency); brainstorming → ~0.8–1.0 with top-p ~0.9 (diversity).

Q · hallucination

“Why do LLMs hallucinate, and how do you reduce it?”

▸ Answer

Because they generate the most plausible next tokens, not the most true, with no built-in “I don’t know” — and confidence is uncorrelated with correctness. Reduce it by grounding in retrieved real data (RAG) with “answer only from context + cite,” giving tools for facts/math, lowering temperature, validating outputs, and showing sources. You manage hallucination with architecture, not by asking the model to be accurate.

Q · base vs instruct

“Base model vs instruct/chat model?”

▸ Answer

A base model is the raw next-token predictor from pre-training — it just continues text. An instruct/chat model is a base model further trained on instruction–response examples and shaped with human preferences (RLHF/DPO), so it follows directions like an assistant. Same engine, extra alignment training.

Q · prompting

“Your classifier returns inconsistent formats. What do you do?”

▸ Answer

Pin the output: add a few-shot examples in the exact format, specify the allowed labels and shape explicitly, and use temperature 0. Inconsistency is unspecified format — specificity narrows the distribution. For machine consumption, use JSON mode + validate + retry.

Q · CoT

“When does chain-of-thought help, and can you trust the reasoning?”

▸ Answer

It helps on multi-step tasks because each written step becomes context the answer builds on, instead of a one-token leap. But the reasoning is generated text — it can look logical and be wrong, and isn’t guaranteed to be the real cause. So use CoT for accuracy but verify high-stakes results (tools, checks, self-consistency); skip it for simple single-step tasks where it just adds cost.

Q · judgment

“Name a task you would NOT hand to an LLM.”

▸ Answer

Anything where a confident wrong answer is costly and the truth isn’t in the prompt — e.g., computing an exact refund from billing rules, or a specific medical/legal determination. The model may produce a fluent, wrong answer with no uncertainty signal. I’d route the exact computation to deterministic code/tools, ground it in authoritative data and verify, or keep a human in the loop. Knowing when not to use an LLM is the senior signal.

An interviewer asks the “one-line” LLM description as a warm-up. Which answer signals the strongest mental model?

  • A) “A next-token predictor that generates text by repeatedly sampling from a probability distribution over its vocabulary.”
    Correct — it leads with the actual mechanism, which is what everything else (cost, limits, hallucination) follows from. This framing lets you answer follow-ups crisply.
  • B) “A database that retrieves the best stored answer.”
    Wrong model — no retrieval of stored answers happens; it computes a distribution from weights. This framing leads to wrong predictions about its behavior.
  • C) “An AI that understands language like a human.”
    Overclaims and obscures the mechanism; it predicts plausible text, which is exactly why it can be confidently wrong. Interviewers want precision, not hype.
  • D) “A reasoning engine that always computes the correct answer.”
    False — it optimizes for plausible, not provably correct, output. This is the opposite of the truth and signals a shaky mental model.
Delivery tips
  • Lead with the mechanism, then the practical consequence (cost / limit / failure mode).
  • Reach for the recap picture: text → tokens → embeddings+attention → next-token odds → sample → repeat.
  • Show judgment: mention when not to use an LLM, and how you’d verify.
  • Keep it tight — 30–45 seconds per answer, then stop and let them follow up.
