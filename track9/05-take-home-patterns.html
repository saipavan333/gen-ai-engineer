Take-home patterns

Many GenAI roles use a take-home: “build a small RAG app,” “make an eval harness,” “create a tool-using agent.” Most candidates make it work and stop there. The ones who get offers add the things that signal a real engineer — and those are exactly what this course taught you.

The common assignments (and what they’re probing)

AssignmentWhat they’re really checking
“Build a Q&A bot over these docs”RAG fundamentals: chunking, retrieval, grounding — and whether you evaluate it
“Improve / evaluate this prompt or system”That you measure with a golden set instead of eyeballing
“Make an agent that does X with tools”Tool calling, the loop, and whether you add guards
“Classify/extract from this dataset”Prompting, structured output, validation, and error handling

What graders actually score

“It runs” is the floor, not the ceiling. Here’s the rubric most reviewers use, explicitly or not:

The winning checklist
  • Make it run from a clean clone. A clear README with exact setup/run steps; no “works on my machine.”
  • Include an evaluation. Even 20 examples with a metric. This is the #1 differentiator — almost nobody does it, and it proves you think like an engineer.
  • Handle failures. Abstain when unsure, validate outputs, don’t crash on weird input. Show you anticipated the messy cases.
  • Justify the architecture. Briefly explain why RAG vs prompt vs agent, why this chunk size, why this model — decisions, not defaults.
  • Note cost/latency. Token counts, where you’d cache, model choice. Even a paragraph shows production awareness.
  • Use the free stack. Everything in this course runs free (Ollama, sentence-transformers, Chroma) — you can ship a complete, runnable take-home at $0.
Worked example — structuring a RAG take-home

A submission that stands out, built entirely from this course’s free tools:

my-rag-takehome/
  README.md          # setup, run, design decisions, trade-offs, "what I'd do next"
  ingest.py          # chunk (recursive + overlap + metadata), embed, store in Chroma
  rag.py             # retrieve → (rerank) → grounded prompt → Ollama; cites sources
  eval.py            # golden set (~25 Q+expected) → recall@k + faithfulness (LLM-judge)
  golden.jsonl       # the eval set — the piece that wins
  tests/             # a couple of regression tests (temp 0, structural asserts)

# README highlights, in order:
#  1) How to run (clone → pip install → python ingest.py → python rag.py)
#  2) Architecture + WHY (RAG because docs are private/changing; chunk size X because…)
#  3) Evaluation results (recall@5 = 0.92, faithfulness = 0.88) and how to reproduce
#  4) Failure handling (abstains when not in context; validates output)
#  5) Cost/latency notes + "what I'd improve with more time"

The eval.py + golden.jsonl are what separate you. Anyone can wire an LLM to a vector store; showing you can measure the result is the rare, hireable signal.

Key points
  • “It works” is the floor; the differentiators are evaluation, edge cases, justified architecture, cost awareness, and a clear README.
  • Include an eval (golden set + metric) — the #1 thing most submissions lack.
  • Explain your decisions and trade-offs in the README — reviewers grade reasoning, not just output.
  • Handle the messy cases (abstain, validate, don’t crash) to show production maturity.
  • You can build a complete, runnable take-home for $0 with this course’s free stack.

You have a working RAG take-home. With limited remaining time, what single addition most improves your chances?

  • A) A small evaluation: a golden set of ~20–30 examples with a metric (recall@k and/or faithfulness) and reproducible results.
    Correct. Evaluation is the rarest and most impressive thing in take-home submissions — it proves you measure quality like an engineer, not by vibes. Most candidates skip it, so adding even a small, honest eval is the highest-leverage differentiator.
  • B) A fancier UI.
    Polish is nice but rarely the deciding factor for an engineering role — reviewers care about correctness, evaluation, and judgment, not visual flair. A pretty UI over an unmeasured system doesn’t signal engineering depth.
  • C) Swapping in the biggest possible model.
    Throwing a bigger model at it shows no engineering judgment and may not even improve results — and it ignores cost. Demonstrating measurement beats demonstrating spend.
  • D) Adding more documents to the index.
    More data doesn’t show you can evaluate or reason about the system. The differentiator is proving quality with an eval, not increasing corpus size.

What most improves a GenAI take-home submission?

  • Including an eval set and honest metrics, plus clear notes on trade-offs and limitations.
    Correct. Evidence and honesty about limits signal an engineer who measures.
  • Maximizing the number of features.
    Depth and evidence beat feature count.
  • Hiding the limitations.
    Stating limitations builds trust; hiding them backfires.
  • Using the newest model for its own sake.
    Justified choices with evidence matter, not novelty.
Practice

1. Your take-home “works” but you’re out of time for a full eval. What’s a minimum-viable eval you can add in 30 minutes?
2. What three things belong in the README that most candidates omit?
3. The prompt says “build a RAG bot.” Why might mentioning what you didn’t do impress the reviewer?

▸ Show solutions & reasoning

1. Hand-write ~15–20 real questions with an expected keyword/answer, then a tiny script that runs your bot and reports how many pass (and, if time, an LLM-judge faithfulness score). It’s not rigorous, but it shows you measure rather than guess — which is the entire point. A small honest eval beats a perfect one you didn’t finish.

2. (a) Why you chose this architecture and key parameters (chunk size, model, RAG vs prompt) — your reasoning. (b) Evaluation results and how to reproduce them. (c) Limitations & “what I’d do with more time” — showing you know what’s missing and how you’d extend it. Most READMEs are just setup steps; these three show engineering judgment.

3. Because it demonstrates judgment and scope awareness: “I used RAG, not fine-tuning, because the docs change; I didn’t build an agent because the task is single-step; I’d add reranking and caching next.” Naming the roads not taken (and why) shows you understand the trade-off space, not just the one path you implemented — which is exactly the senior signal reviewers look for.

Deep dive: a take-home is a portfolio piece

Reframe the take-home: it’s not a hoop, it’s the start of your portfolio. The same project that wins one company’s take-home becomes your GitHub showcase, your answer in the next system-design round, and the source of your behavioral stories — the “one strong project” strategy in action. So invest in making it genuinely good and legible: clean structure, an honest README, real (if small) evaluation, and a clear narrative of decisions. A reviewer who can read your reasoning in five minutes and see that you measured, handled failures, and understood the trade-offs will rank you far above a more complex submission they can’t follow or that’s unmeasured.

A few practical do’s and don’ts. Do scope down and finish — a small, complete, evaluated, well-documented project beats an ambitious, broken one. Do use the free stack so it runs for any reviewer at $0 (Ollama + sentence-transformers + Chroma). Don’t over-engineer (a five-agent system where a prompt would do signals poor judgment, exactly the opposite of what they want). Don’t hide your assumptions — state them.

And do include the unglamorous things that prove maturity: error handling, an eval, cost notes, and a “limitations” section. The meta-lesson of this whole course shows up here in miniature: anyone can make an LLM produce output; the engineer is the one who can measure and trust it — and your take-home is where you prove it.
