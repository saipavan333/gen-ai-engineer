Track 15 · Lesson 15.3

Agents Q&A

Agentic system-design questions are increasingly common: design something where an LLM uses tools to reason, plan, and act. Interviewers want maturity — you give a model autonomy responsibly. This bank covers the questions, with the security and “when not to” judgment that signals seniority.

What they’re really testing

Whether you can reason about when to use tools vs. pure generation, planning vs. direct tool calls, and — above all — how you keep an autonomous, tool-using system safe and bounded. The strongest signal: reaching for the least autonomy that solves the task.

The bank

Q · framing

“When do you use tools vs. pure text generation, and when do you build an agent at all?”

▸ Answer

Use a tool when the task needs something the model can’t do reliably from text — fresh data, exact computation, or an action in the world. Build an agent (a loop) only when the path depends on what each step reveals; for fixed, single-step or known-pipeline tasks, a plain prompt or RAG is simpler and more reliable. Principle: the least autonomy that solves the task — agents add cost and failure surface, so I justify them.

Q · tool calling

“How does tool/function calling work end to end?”

▸ Answer

You expose tools with schemas (name, description, typed params). The model emits a structured tool call (which tool, what args) — it never runs code itself. Your code executes the function and returns the result as a tool-role message; the model continues. Tool descriptions are prompt surface (clear ones drive correct selection), and arguments are untrusted input you validate.

Q · the loop

“Explain the agent loop and why it needs a step limit.”

▸ Answer

The loop repeatedly asks the model what to do: call a tool (whose observation is appended) or emit a final answer. ReAct interleaves reasoning with acting so each step builds on the last. It needs a max-step cap because the model decides when to stop, and a confused model can loop forever — the cap guarantees termination and bounds cost. I also log the full trace; agent debugging is reading traces.

Q · planning

“Planning-based agents vs. direct tool calls — when each?”

▸ Answer

Direct (reactive) tool calls suit short tasks where each step reveals the next. Planning — decompose the goal into steps first — suits longer, open-ended goals: you get an inspectable, approvable plan and possible parallelism. For surprise-prone tasks I prefer plan-and-reflect (re-check after each step and replan) over executing a fixed plan blindly. Best agents combine planning (structure) with reactive execution (adaptivity).

Q · memory

“How do agents remember things across steps and sessions?”

▸ Answer

Two layers: short-term is the context window (summarize old turns to fit), long-term is an external store you write to and retrieve from — which is just RAG over the agent’s own history (episodic events + semantic facts). The hard part is curation, not storage: extract salient facts, dedupe/update them, and retrieve precisely; and treat the memory store as sensitive data (privacy).

Q · MCP

“What is MCP and why does it matter?”

▸ Answer

An open standard for connecting AI apps to tools/data — “USB-C for AI.” A client connects to servers exposing tools, resources, and prompts through one protocol, so you build a connector once and any compatible agent uses it. It’s become industry-standard infrastructure, which shifts agent-building toward connecting trusted servers — and makes server trust a real security question.

Q · multi-agent

“When would you use multiple agents, and what’s the risk?”

▸ Answer

Use specialists + an orchestrator when roles are genuinely separable (router, pipeline, parallel). The risk is coordination overhead — lossy handoffs, compounding errors, higher cost and harder debugging. So single-agent-first: one well-tooled agent usually beats five poorly-coordinated ones; add agents only when a single agent measurably struggles.

Q · safety (the big one)

“How do you make an autonomous, tool-using agent safe for production?”

▸ Answer

Defense in depth: bound the loop (max steps), validate tool arguments, cap cost/time (budgets, timeouts), require human approval for irreversible/high-impact actions, apply least privilege (only the tools the task needs), sandbox dangerous capabilities, and log every step. Hard limits live in code, not the prompt (injection can override prompts). Treat tool/retrieved content as untrusted (injection risk). And evaluate the agent on representative tasks on every change, since I can’t eyeball dozens of autonomous actions.

“Design an agent that can issue refunds.” Which detail most signals seniority?

  • A) A hard, code-level refund cap plus human approval above a threshold — never trusting the prompt for the limit, since injection can override it.
    Correct. Recognizing that the dangerous, irreversible action must be constrained in the trusted layer (code) — not the manipulable prompt — and gated by a human, shows you understand agent security and containment. It’s the detail that separates “I built an agent” from “I built one I’d let touch money.”
  • B) Using the biggest available model for accuracy.
    A bigger model is still injectable and, with an unrestricted refund tool, can still be tricked into harm. Capability doesn’t contain impact; constraints and approval do.
  • C) A longer system prompt forbidding large refunds.
    Prompt instructions can be overridden by injection and aren’t enforced. The limit must be in code, outside the model’s influence.
  • D) Higher temperature for flexible responses.
    Irrelevant to safety and makes behavior less predictable. The senior detail is containment of the money-moving action.
Delivery tips
  • State the least-autonomy principle early — show you don’t reach for agents reflexively.
  • For any action-taking agent, volunteer the guard stack (least privilege, code-level limits, human approval, validation, monitoring).
  • Treat tool/retrieved content as untrusted (injection) — security awareness is a major signal.
  • Mention evaluating agents on task suites — you can’t eyeball autonomous behavior.
