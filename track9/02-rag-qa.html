Track 9 · Lesson 9.2

RAG Q&A

RAG is the single most heavily-tested topic in applied-GenAI interviews this year — end-to-end design, chunking, evaluation, and production trade-offs. This bank covers the questions teams actually ask, including the advanced ones that separate strong candidates.

What they’re really testing

That you treat RAG as a retrieval + grounding + evaluation system, can separate retrieval failures from generation failures, and reason about cost, latency, and scale — not just recite the pipeline. The recurring tell of a senior answer: “I’d measure which half is failing first.”

The bank

Q · pipeline

“Walk me through a production RAG pipeline.”

▸ Answer

Offline: ingest → chunk (with overlap + metadata) → embed → store vectors. Online per query: embed the query (same model) → retrieve top candidates (often hybrid dense+keyword) → rerank with a cross-encoder → assemble a grounded prompt with numbered sources → generate with “answer only from context + cite.” Guard each stage with metrics: recall@k/MRR for retrieval, faithfulness/correctness for generation.

Q · chunking

“Compare fixed-size, recursive, and semantic chunking. How do you pick a size?”

▸ Answer

Fixed-size cuts every N tokens (simple baseline); recursive/structural respects paragraphs/headings/code (the sensible default); semantic splits where the topic shifts (best coherence, more compute). Aim for one coherent idea per chunk (~200–500 tokens) with overlap so boundary ideas survive, and attach metadata for citations/filtering. Crucially, I tune size with evaluation — it’s a recall@k knob, not a guess — and ensure chunks fit the embedding model’s max length.

Q · embeddings

“What’s the most common embedding bug?”

▸ Answer

Embedding documents and queries with different models (or changing the model without re-indexing) — the vectors become incomparable and retrieval silently breaks. Same model both sides, normalize so dot = cosine, re-embed everything on a model change. And pick the model empirically (recall@k on your data), respecting max sequence length and any query/passage prefix convention.

Q · reranking

“Why add a reranker, and when does it not help?”

▸ Answer

A bi-encoder embeds query and doc separately (fast, blunt); a cross-encoder reads them together (accurate, slow). Two-stage retrieval fetches many candidates cheaply then reranks the shortlist for precision — often a bigger quality win than a larger LLM. It only helps if recall is already good: reranking reorders the candidate set but can’t surface a chunk retrieval never fetched. If recall@k is low, fix retrieval first.

Q · hybrid & query transforms

“When is semantic search not enough, and what’s query transformation?”

▸ Answer

Pure dense search misses exact strings — codes, SKUs, names — so I add keyword (BM25) search and fuse with RRF (hybrid). On the query side, query transformation improves retrieval: rewriting a vague query, decomposing a multi-part question into sub-queries, step-back prompting (ask a more general question first), or HyDE (generate a hypothetical answer and embed that to retrieve). These help when the raw query embeds poorly relative to the documents.

Q · scale

“How do you scale RAG to millions of documents?”

▸ Answer

Use a real vector database with an ANN index (HNSW/IVF) so search is sub-linear, tuned to a recall/latency target (ef/nprobe). Add metadata filtering (and per-tenant isolation for multi-tenancy — a security control). Shard/replicate for throughput, quantize vectors to cut storage, cache frequent queries, and keep a two-stage retrieve→rerank so the expensive step only runs on a shortlist. And measure recall on a held-out set at scale — ANN trades a little recall for big speed.

Q · grounding

“How do you make answers trustworthy and citable?”

▸ Answer

Number the retrieved chunks, instruct the model to answer only from them, cite [n] per claim, and say “not in the documents” when absent. Then verify citations programmatically (the cited chunk exists and actually supports the claim — overlap or LLM-judge faithfulness), and surface sources for high-stakes use. The goal isn’t a trustworthy model; it’s a system where every claim is checkable.

Q · evaluation

“How do you evaluate a RAG system?”

▸ Answer

Measure the two stages separately on a golden set: retrieval with recall@k and MRR ((query → relevant chunk) labels), generation with faithfulness (claims supported by context) and answer correctness (via LLM-as-judge). Run it offline as a CI gate, trace production to mine new failures into the golden set, and A/B online watching quality plus latency/cost. Separating retrieval from generation tells me which half to fix.

Q · debugging

“A RAG answer is wrong. How do you debug it?”

▸ Answer

Open the trace and check retrieval for that query: was the relevant chunk in the top-k? If not, it’s retrieval — look at chunking, embedding model, hybrid search, query transformation, or fetch-more-then-rerank. If it was retrieved but the answer ignored or contradicted it, it’s generation — check grounding/faithfulness and the prompt. Reproduce with the exact retrieved context. The recall@k number for that query tells me which half to fix — never guess.

An interviewer says: “Recall@10 is 0.97 but users still get wrong answers.” The senior response is…

  • A) Retrieval is healthy, so the problem is generation/grounding — check faithfulness; the model is ignoring or misusing the retrieved context.
    Correct. High recall means the right chunks reach the prompt, so a wrong answer points downstream: weak grounding, low faithfulness, or the answer not actually using the context. You’d measure faithfulness and tighten the grounding prompt — not touch retrieval.
  • B) Improve the embedding model.
    Recall@10 = 0.97 says retrieval (and the embeddings) are working. Changing the embedder targets a non-problem and ignores the generation signal.
  • C) Increase k to 50.
    More chunks won’t make the model use them faithfully and can worsen “lost in the middle.” The diagnostic points at generation, not retrieval breadth.
  • D) Re-chunk the corpus.
    Chunking affects retrieval, which is already strong. The fix lives in the generation/grounding stage, which separating the metrics reveals.
Delivery tips
  • Always separate retrieval vs generation — it’s the answer to half of RAG questions.
  • Mention evaluation unprompted (recall@k/MRR + faithfulness) — it signals production maturity.
  • Bring up cost/latency/scale trade-offs (ANN, hybrid, rerank-the-shortlist) — interviewers want these.
  • Know the advanced levers (hybrid, reranking, query transformation/HyDE) for the “how would you improve it?” follow-up.
