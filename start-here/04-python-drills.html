Drill 1 — Filter a list (comprehension)

Drill 1

Start from words = "RAG gives a model grounded knowledge".split(). Build a new list containing only the words longer than 4 characters.

▸ Show solution & reasoning 
words = "RAG gives a model grounded knowledge".split()
long_words = [w for w in words if len(w) > 4]
print(long_words)   # -> ['gives', 'model', 'grounded', 'knowledge']

Why: .split() turns a sentence into a list of words; the trailing if keeps only those passing the test. This exact shape — “keep the items that qualify” — is how you’ll drop empty or too-short chunks before sending them to a model.

Drill 2 — Count things (frequency dict)

Drill 2

Given intents = ["refund", "refund", "shipping", "refund", "returns", "shipping"], build a dictionary that maps each intent to how many times it appears.

▸ Show solution & reasoning 
intents = ["refund", "refund", "shipping", "refund", "returns", "shipping"]

counts = {}
for i in intents:
    counts[i] = counts.get(i, 0) + 1

print(counts)   # -> {'refund': 3, 'shipping': 2, 'returns': 1}

Why: the counts.get(i, 0) move reads the current tally (or 0 if this intent is new), adds one, and stores it back. Counting is everywhere in evals — “how many answers were correct,” “how often did each tool get called” — and this is the canonical way to do it.

Drill 3 — Build a messages list

Drill 3

Write code that, given system = "You are concise." and turns = ["Hi", "What is a token?"], builds the chat messages list: a system dict first, then one user dict per turn.

▸ Show solution & reasoning 
system = "You are concise."
turns = ["Hi", "What is a token?"]

messages = [{"role": "system", "content": system}]
for t in turns:
    messages.append({"role": "user", "content": t})

print(messages)
# [{'role': 'system', 'content': 'You are concise.'},
#  {'role': 'user', 'content': 'Hi'},
#  {'role': 'user', 'content': 'What is a token?'}]

Why: this is the literal payload every chat model wants — a list of role/content dicts, system first. You just built, by hand, the structure that libraries hide behind a chat() call. Seeing it plainly now means none of the later code is mysterious.

Drill 4 — A tiny chunker

Drill 4

Write chunk(text, size) that splits a string into a list of pieces, each at most size characters long. For example, chunk("abcdefg", 3) should give ["abc", "def", "g"].

▸ Show solution & reasoning 
def chunk(text, size):
    return [text[i:i + size] for i in range(0, len(text), size)]

print(chunk("abcdefg", 3))   # -> ['abc', 'def', 'g']

Why: range(0, len(text), size) steps in jumps of size (0, 3, 6…), and text[i:i+size] slices that piece. You just wrote the simplest possible document chunker — the very thing Track 3 builds on (real chunkers add overlap and split on sentence boundaries, but this is the skeleton).

Drill 5 — Read a nested field safely

Drill 5

A model reply looks like {"text": "...", "usage": {"total_tokens": 42}} — but sometimes the "usage" block is missing entirely. Write total_tokens(reply) that returns the token count, or 0 if usage isn’t there.

▸ Show solution & reasoning 
def total_tokens(reply):
    return reply.get("usage", {}).get("total_tokens", 0)

print(total_tokens({"usage": {"total_tokens": 42}}))   # -> 42
print(total_tokens({"text": "hi"}))                     # -> 0

Why: the trick is the empty-dict default. reply.get("usage", {}) hands back a real (empty) dict when usage is missing, so the second .get still works instead of crashing. Chaining .get with a {} fallback is the clean way to dig into nested, unreliable API responses — and you’ll read token usage exactly like this in Track 8.

Drill 6 — Keyword search (the seed of retrieval)

Drill 6

Write search(query, docs) that returns every document containing all of the query’s words (case-insensitive). Test it with query = "refund time" against a few sentences.

▸ Show solution & reasoning 
def search(query, docs):
    terms = query.lower().split()
    return [d for d in docs if all(t in d.lower() for t in terms)]

docs = [
    "Refund time is 5 business days.",
    "Shipping is free over $50.",
    "You can request a refund any time within 30 days.",
]
for hit in search("refund time", docs):
    print(hit)
# Refund time is 5 business days.
# You can request a refund any time within 30 days.

Why: all(...) is true only when every term is present, and the lowercasing makes the match case-insensitive. This is keyword (lexical) search in miniature — the “sparse” half of the hybrid search you’ll build in Track 3. You’ve now hand-written both a chunker and a retriever; RAG will feel like assembling parts you already understand.

In Drill 5, why is the default in reply.get("usage", {}) an empty dict {} rather than 0 or None?

  • A) So the next .get("total_tokens", 0) still has a dict to call it on, even when usage is missing.
    Correct. The whole point is to keep chaining. If usage is absent, {} is returned — a real dict — so .get("total_tokens", 0) runs safely and yields 0. Return None or 0 instead and that second .get would crash with an AttributeError.
  • B) Because {} is faster than 0.
    Speed isn’t the issue at all. The default is about type: the next step needs something it can call .get on, i.e. a dict.
  • C) Because token counts are always stored as dictionaries.
    The token count itself is a number; usage is the dict that holds it. The {} default stands in for a missing usage block, not for the count.
  • D) It doesn’t matter — any default works the same.
    It matters a lot. Only a dict default lets the chained .get survive. Try it with None and you’ll get AttributeError: 'NoneType' object has no attribute 'get'.

Deep dive: you already have the muscles for this

If you’re coming from data engineering, notice what these drills actually are: filtering rows, grouping and counting, reshaping records, and a lexical search. You’ve done every one of these a thousand times in SQL and pandas — WHERE is Drill 1, GROUP BY ... COUNT is Drill 2, building a JSON payload is Drill 3, and a LIKE '%term%' filter is Drill 6. The syntax is new; the thinking is not. Much of applied GenAI is data wrangling wearing a different hat, which is exactly why a data background transfers so well. Don’t let unfamiliar libraries convince you you’re a beginner here — you’re a practitioner learning a new dialect.
