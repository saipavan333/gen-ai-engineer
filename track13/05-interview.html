Interview check: Data & SQL agents

Text-to-SQL is a favorite interview topic precisely because it exposes whether you think about safety, scale, and correctness — not just prompting. It’s also where your data background lets you outshine everyone. Here are the questions teams ask, with strong answers.

How to use this

Answer each one out loud before opening the model answer. This track’s bar: could you design a trustworthy “chat with your database” feature and defend every safety and correctness decision? If yes, you can hold your own in the rounds that most reward a data engineer.

The questions teams ask

Q1 · Warm-up

“Walk me through how a text-to-SQL system works, end to end.”

▸ Strong answer & what they’re testing

Answer: A user asks in plain English. I build a prompt containing the relevant schema (tables, columns, relationships, a few sample rows and example query pairs) plus the question, and the model returns a SQL query. I validate it (single SELECT), run it through a read-only connection with a row limit, and turn the rows into a worded answer — returning the SQL too, for transparency. If the query errors, I feed the database’s error message back and let the model revise, up to a small retry cap.

Testing: Whether you have the full pipeline, including safety (read-only) and reliability (self-correction), rather than just “ask the model for SQL and run it.”

Q2 · Safety (they will push here)

“The model could generate DROP TABLE. How do you prevent damage?”

▸ Strong answer & what they’re testing

Answer: Defense in depth, with a read-only database user as the bedrock — if the credentials can’t write, no generated query can delete or modify anything, and no prompt injection can change that. On top I validate the query is a single SELECT/WITH and reject stacked statements, force a LIMIT, set a statement timeout, and parameterize any user-supplied values. I never rely on the prompt (“please only SELECT”) as a control — that’s a hint, not a wall. Permissions are the wall.

Testing: Least-privilege thinking. The phrase they want is “read-only database user”; leading with a prompt instruction is a red flag.

Q3 · Scale

“Our warehouse has 500 tables. They can’t all fit in the prompt. Now what?”

▸ Strong answer & what they’re testing

Answer: Schema retrieval — RAG over metadata. I embed each table’s name and description, and for a given question retrieve just the handful of relevant tables to put in the prompt. That keeps context small, cheap, and focused, and it improves accuracy by not drowning the model in irrelevant tables. I’d also invest in a curated semantic layer — clean views, a data dictionary, canonical metric definitions — so the model queries a friendly, well-documented surface instead of raw production tables.

Testing: That you connect this to RAG (Track 3) and that you know accuracy comes from the semantic layer, not a bigger model.

Q4 · The big one: evaluation

“How do you know your text-to-SQL system is actually correct?”

▸ Strong answer & what they’re testing

Answer: I build a golden set of question → correct-SQL pairs across easy and hard cases, then measure execution accuracy: run the generated query and the reference query and compare their result sets — not the SQL strings. String matching is wrong because many different queries are equally correct (different join order, aliases, subquery vs CTE); what matters is whether they return the same rows.

I track accuracy overall and by difficulty, watch specific failure modes (bad joins, wrong date boundaries, ambiguity), and wire it into CI so a prompt or schema change can’t silently regress. In production I also log queries, add user feedback (“was this right?”), and monitor error/retry rates.

Testing: This is the senior differentiator. The key insights: compare results, not query text; use a golden set; and evaluate continuously. It’s Track 6 applied to SQL.

Q5 · Reliability

“The generated query fails or returns something odd. How does your system recover?”

▸ Strong answer & what they’re testing

Answer: On an error, I feed the database’s exact error message back to the model and let it revise, within a bounded retry loop (2–3 tries) — the ReAct pattern; the read-only runner means retrying is harmless. For “runs but looks wrong,” I sanity-check the result shape (a “how many” question shouldn’t return 40,000 rows) and can feed that mismatch back too. And for genuinely ambiguous questions, I’d rather have the system ask a clarifying question than confidently guess.

Testing: Whether you know self-correction (feed the error back), bounded loops, and that asking-when-unsure beats guessing.

Q6 · System design

“Design a ‘chat with your database’ feature for non-technical staff.”

▸ Strong answer & what they’re testing

Answer: Clarify first: which database, how many tables, read-only analytics or actions, latency and accuracy expectations, who the users are. Then: a curated semantic layer (clean views + dictionary + example queries); schema retrieval to select relevant tables per question; the model generates SQL at temperature 0; a safety layer (read-only user, SELECT-only validation, LIMIT, timeout); a self-correcting retry loop; and an answer that shows the SQL and a result preview so users can trust it. Around it: a golden-set eval in CI, query logging, feedback capture, and cost/latency monitoring. I’d start narrow — a few well-modeled tables — and expand as evaluation shows it’s reliable.

Testing: Structure, safety, evaluation, and pragmatism (start narrow). Clarifying before designing is a senior tell.

Q7 · Judgment

“When would you not ship text-to-SQL, or heavily constrain it?”

▸ Strong answer & what they’re testing

Answer: When wrong answers are costly and hard to notice — regulated financial or medical reporting, say — free-form generated SQL is risky because of the confident-wrong-number problem. There I’d constrain hard: restrict to a small set of validated, parameterized queries or a curated metrics layer rather than open-ended generation, keep a human in the loop for anything consequential, and always surface the SQL. The skill is matching how much freedom the model gets to how costly a wrong answer is.

Testing: Maturity — knowing that “generate any SQL” isn’t always appropriate, and how to dial down the model’s freedom by risk.

Rapid-fire self-check

What’s the right way to measure whether a generated SQL query is “correct” against a reference query?

  • A) Run both and compare their result sets (execution accuracy) — different SQL can be equally correct.
    Correct. Execution accuracy compares what the queries return, which is what matters. Many correct queries differ in text — join order, aliases, CTE vs subquery — so comparing result sets (order-insensitively) is the valid measure.
  • B) Check that the generated SQL string exactly matches the reference SQL.
    String matching fails constantly: two textually different queries can be identically correct. Exact-match would mark correct answers as wrong and is the classic mistake in text-to-SQL evaluation.
  • C) Ask the model whether its own query is correct.
    Self-assessment isn’t ground truth — the same misunderstanding that produced a wrong query can rate it “correct.” You need an objective check against known-good results.
  • D) Count the number of tables the query joins.
    Join count says nothing about correctness — a correct query might join one table or five. It’s unrelated to whether the result is right.

An interviewer asks for the single most important safety control in a text-to-SQL system. Best answer?

  • A) Run queries through a read-only database user, so writes are impossible regardless of what’s generated.
    Correct. Least privilege at the database level is the one control an attacker or a bad generation can’t bypass — the database itself refuses to write. Everything else layers on top of this bedrock.
  • B) A system prompt that says “never write destructive SQL.”
    A prompt is a hint, not a control — prompt injection can override it. Relying on it as your main safety is the cardinal sin of text-to-SQL.
  • C) Only allowing the model to use small tables.
    Table size is unrelated to write-safety; a destructive query on a small table is still destructive. This doesn’t address the actual risk.
  • D) Rate-limiting how many queries a user can run.
    Rate limits help with load and abuse volume, but a single DROP TABLE within the limit still causes damage. It’s not the control that prevents destruction.

How do you stop a text-to-SQL agent from ever running a destructive statement like DELETE?

  • Parse the SQL and allowlist statement types (SELECT only) in code, rejecting everything else.
    Correct. An allowlist on the parsed statement type fails closed and can’t be talked around, unlike blocklisting dangerous words or trusting the prompt.
  • Tell the model in the prompt never to write DELETE.
    A prompt is manipulable; the guarantee must be in code.
  • Scan the output text for the word ‘delete’.
    Blocklists are endless to maintain and trivially evaded (comments, casing, second statements).
  • Use a larger model.
    Capability doesn’t enforce a safety boundary.

The generated SQL passes your safety gate and ‘looks right’. What can still be wrong that the gate won’t catch?

  • Correctness — a wrong join, date window, or status value; safety and correctness are different problems.
    Correct. The gate proves the query is safe to run, not that it answers the question. Correctness needs evals and, sometimes, a human.
  • Nothing — passing the gate means it’s correct.
    Safe ≠ correct; the gate only checks it’s read-only and allowlisted.
  • It will always delete data.
    A SELECT-only gate prevents writes entirely.
  • It will crash the database.
    A validated read-only query with a row limit won’t.
Can you explain each of these? (the real test)
  • The end-to-end pipeline: schema prompt → SQL → safe run → answer, with self-correction.
  • Why the read-only user is the bedrock safety control, and the layers on top.
  • Scaling with schema retrieval and a semantic layer.
  • Execution accuracy on a golden set — comparing results, not SQL strings — in CI.
  • The self-correction loop, bounded retries, and asking when a question is ambiguous.
  • When to constrain or not ship open-ended SQL by how costly a wrong answer is.
