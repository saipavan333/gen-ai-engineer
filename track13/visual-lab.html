Watch a question become safe SQL

Text-to-SQL is the highest-leverage GenAI skill for a data engineer — and the one where a wrong move costs the most. Watch a plain-English question become a query, pass a safety gate, and run; then watch a destructive request meet the same gate.

How to use this lab

Press ▶ Play for the whole journey, or Step to pause on each stage. Watch the safety gate at stage 4 — and what it does at stage 6 when the request turns destructive.

Make it yours

1. The gate rejects the DELETE by statement type, not by scanning for the word "delete". Why is checking the parsed statement type safer than string-matching for dangerous words?
2. The generated SELECT "looks right". Name two things that could still be wrong with it that the safety gate would happily let through.

▸ Show reasoning

1. String-matching is a blocklist, and blocklists are endless to maintain and trivial to evade — comments, casing, encodings, a DELETE hidden after a semicolon in a second statement. Parsing the SQL and asking "what statement type is this?" is an allowlist: you permit exactly SELECT and reject everything else by construction, including forms you never thought to blocklist. Allowlists fail closed; blocklists fail open.

2. The gate guarantees the query is safe to run, not correct. It could still join on the wrong key, use the wrong date window, double-count from a fan-out join, or filter on status = 'cancelled' when the real value is 'canceled'. Safety and correctness are different problems — the gate handles the first; evals and a human handle the second.

What you just saw
  • Schema grounding — the model needs the real tables and columns, not a guess.
  • Generated SQL that looks right, which is exactly what you must not trust.
  • A safety gate in code — allowlist, read-only, single statement, row limit.
  • The same gate rejecting a destructive request the model proposed.
