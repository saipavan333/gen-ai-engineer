Structured Data & Text-to-SQL · Lesson 13.1

Why text-to-SQL (and where it breaks)

Most of the world’s valuable data lives in databases, and most of the people who need answers can’t write SQL. Text-to-SQL closes that gap — and it’s the single most natural place for a data engineer to shine in GenAI. This lesson frames the problem, the payoff, and the honest failure modes.

The problem it kills

In every company there’s a queue: a marketer wants “revenue by region last quarter,” a support lead wants “tickets reopened more than twice,” an exec wants “top 10 customers by lifetime value.” Each of these is a five-minute SQL query — but only for the handful of people who know SQL and the schema. Everyone else waits on that bottleneck. Text-to-SQL is a system that takes a plain-English question, writes the SQL for it, runs it against the database, and returns the answer in words. It turns “ask the data team and wait two days” into “ask and get an answer now.”

This is enormous because it applies almost everywhere: analytics assistants, internal “chat with your database” tools, customer-facing reporting, and BI copilots are among the most-funded, most-built GenAI products right now. And — importantly for you — it’s a problem where knowing databases is the hard part, not knowing the LLM. That’s your home field.

What it looks like

Imagine a tiny sales database — the kind you’ll actually build in this track with SQLite (it’s built into Python, so there’s nothing to install and nothing to pay for). Two tables:

schema.sql
CREATE TABLE customers (
    id       INTEGER PRIMARY KEY,
    name     TEXT,
    country  TEXT,
    signup   DATE
);
CREATE TABLE orders (
    id           INTEGER PRIMARY KEY,
    customer_id  INTEGER,
    amount       REAL,
    created_at   DATE
);

A user asks: “How much did customers in Canada spend in total?” A good text-to-SQL system, given that schema, produces:

generated.sql
SELECT SUM(o.amount) AS total_spent
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE c.country = 'Canada';

…runs it, gets [(48210.5,)], and answers: “Customers in Canada have spent $48,210.50 in total.” The magic isn’t the English — it’s that the model had to understand the schema (which table has amounts, how to join customers to orders, that country lives on customers) to write correct SQL. Schema understanding is the whole game, and it’s exactly the next lesson.

Where it breaks (be honest in interviews)

Text-to-SQL demos beautifully and then meets reality. The failure modes are predictable, and naming them is a senior signal:

Ambiguity. “Show me our best customers” — best by revenue? order count? recency? The model has to guess, and may guess differently than the user meant. Schema misunderstanding. It may hallucinate a column that doesn’t exist, pick the wrong table when names are similar, or botch a multi-table join. Wrong-but-runs. The scariest case: the SQL executes without error but computes the wrong thing (e.g., double-counts through a bad join), returning a confident, wrong number. Scale & dialect. Hundreds of tables won’t fit in the prompt, and SQL dialects differ (SQLite vs Postgres vs BigQuery). Each of these has a mitigation you’ll learn — schema selection, validation, self-correction, and evaluation — but a system that ignores them is a liability, because a wrong number that looks right is worse than an error.

Key points
  • Text-to-SQL = English question → generated SQL → safe run → worded answer. It removes the analyst bottleneck.
  • It’s one of the highest-demand GenAI applications, and the hard part is databases — your advantage.
  • Correctness hinges on the model understanding the schema, not on fancy language.
  • Failure modes: ambiguity, schema misunderstanding, wrong-but-runs, scale/dialect. Each has a mitigation.
  • The dangerous failure is a confident wrong number — which is why validation and evaluation matter as much as generation.
Practice

For the two-table schema above, write (by hand) the SQL you’d want the model to produce for: “How many orders did we get in 2025?” Then name one way a text-to-SQL system could get this specific question subtly wrong.

▸ Show solution & reasoning 
SELECT COUNT(*) AS order_count
FROM orders
WHERE created_at >= '2025-01-01'
  AND created_at < '2026-01-01';

A subtle way it goes wrong: the model might write WHERE strftime('%Y', created_at) = '2025' — correct in SQLite but broken in other dialects — or, worse, use created_at <= '2025-12-31', which silently drops orders with a time component late on Dec 31 (since '2025-12-31 14:00' > '2025-12-31'). Both run and return a plausible count, so no error surfaces — this is the “wrong-but-runs” trap. The half-open range >= start AND < next-year is the robust pattern, and noticing this is exactly the kind of data-engineering judgment that makes your answers stand out.

Why is a text-to-SQL query that runs successfully but returns the wrong number considered more dangerous than one that throws an error?

  • A) An error is visible and gets fixed; a wrong-but-plausible number is trusted and acted on.
    Correct. Errors are self-announcing — someone sees them and investigates. A query that silently double-counts or misfilters returns a confident number that looks right, so it flows into a decision or a report unquestioned. Undetected wrong answers erode trust in the whole system, which is why validation and evaluation (later lessons) matter as much as generation.
  • B) Errors crash the database, while wrong numbers don’t.
    A failed SELECT doesn’t crash a database — it just returns an error to the caller. The danger of the wrong number isn’t about stability; it’s that no one knows it’s wrong.
  • C) Wrong numbers are always caused by SQL injection.
    Injection is a separate security concern. Wrong-but-runs errors usually come from bad joins, ambiguous questions, or filter mistakes — logic problems, not attacks.
  • D) They aren’t more dangerous; both are equally easy to catch.
    They’re not equally catchable — that’s the whole point. An error is caught immediately; a plausible wrong number can go unnoticed indefinitely, which makes it the more dangerous outcome.
Deep dive: why this is your unfair advantage

Text-to-SQL sits precisely at the intersection of LLMs and data engineering, and the LLM half is the easy half. Anyone can call a model and ask for SQL; almost no one applying for these roles can reason about join fan-out, null handling, date-boundary bugs, dialect differences, indexing, or how to give a model just the relevant slice of a 400-table warehouse without blowing the context window. Those are the things that decide whether the generated SQL is correct at scale — and they’re your bread and butter from a decade of pipelines.

There’s also a strategic idea worth carrying into interviews: the best text-to-SQL systems don’t point the model at raw tables. They put a semantic layer in between — curated views, clear column names, documented metrics (“revenue means SUM(amount) on paid orders”), and example query pairs. That turns an ambiguous natural-language problem into a constrained, well-labeled one the model rarely gets wrong. Designing that layer is data modeling, which is exactly what you already do. So when you frame text-to-SQL as “a data-modeling and correctness problem with an LLM on top,” you’re not learning a new field — you’re applying your field to a new interface.

The whole system rises or falls on how well the model understands your schema. So that’s where we start building. Next: schema-aware prompting.
