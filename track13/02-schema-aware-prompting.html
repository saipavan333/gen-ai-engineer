Structured Data & Text-to-SQL · Lesson 9.2

Schema-aware prompting

A text-to-SQL model is only as good as its picture of your database. This lesson is about feeding it the right context — the schema, descriptions, and a few examples — and, when you have hundreds of tables, giving it just the slice it needs.

Why the schema is the prompt

A model cannot query columns it doesn’t know exist. Ask “total spend in Canada” with no schema and it will guess table and column names — sometimes right, often not. Give it the exact schema and the same question becomes almost mechanical. So the core of every text-to-SQL prompt is a clear description of the tables involved: their columns, types, and how they relate. The better and cleaner that description, the more the problem shifts from “creative writing” to “fill in the blanks,” which is where models are reliable.

The minimum viable schema context is the CREATE TABLE statements. Better still, add a one-line description per table and per non-obvious column, and — this helps a lot — a couple of sample rows, so the model sees that country holds 'Canada' not 'CA', and that status is 'paid' not 1.

build_schema_prompt.py
def schema_context():
    return """
Table customers(id, name, country, signup DATE)
  -- one row per customer. country is a full name, e.g. 'Canada'.
Table orders(id, customer_id, amount REAL, status TEXT, created_at DATE)
  -- one row per order. status is 'paid' or 'refunded'. amount is in USD.
  -- orders.customer_id joins to customers.id
Sample: customers -> (1, 'Ada Lovelace', 'Canada', '2024-02-11')
Sample: orders    -> (55, 1, 120.00, 'paid', '2025-06-03')
"""

def build_prompt(question):
    return f"""You write SQLite SQL. Use ONLY the tables and columns below.
Return a single SELECT query and nothing else.

{schema_context()}

Question: {question}
SQL:"""

Notice three instructions doing quiet, important work: “use ONLY the tables and columns below” (fights hallucinated columns), “a single SELECT” (a first safety rail — more in the next lesson), and naming the dialect “SQLite” (so it doesn’t reach for Postgres-only functions).

Show, don’t just tell: few-shot examples

The most reliable upgrade is adding a few example pairs — a question and its correct SQL — right in the prompt. This is few-shot prompting (Track 2) applied to SQL, and it teaches the model your conventions: how you compute “revenue,” that you always filter status = 'paid', how you handle dates. Two or three well-chosen examples often do more for accuracy than any amount of instruction.

examples.txt
Q: How many paid orders were there in 2025?
A: SELECT COUNT(*) FROM orders
   WHERE status = 'paid'
     AND created_at >= '2025-01-01' AND created_at < '2026-01-01';

Q: Total revenue by country.
A: SELECT c.country, SUM(o.amount) AS revenue
   FROM orders o JOIN customers c ON c.id = o.customer_id
   WHERE o.status = 'paid'
   GROUP BY c.country;

When there are 400 tables: select, don’t dump

Real warehouses have hundreds of tables — far more than fit in a prompt, and dumping them all would drown the model in irrelevant options (and cost a fortune in tokens). The fix is schema retrieval, and it’s exactly RAG (Track 3) pointed at your metadata: embed each table’s name + description, and for a given question retrieve only the handful of tables most likely relevant, then build the prompt from just those. A question about refunds pulls in orders and refunds, not marketing_emails. You’re using retrieval to keep the context small, focused, and cheap — the same discipline as document RAG, applied to schema.

This is RAG you already understand

Schema selection is just retrieval with tables as the “documents.” Everything you learned in Track 3 — embeddings, top-k, keeping context tight — transfers directly. Reusing that muscle is why RAG comes before this track.

Key points
  • The schema is the prompt: give exact tables, columns, types, relationships — plus short descriptions and a few sample rows.
  • Add instructions that constrain: “use ONLY these columns,” “single SELECT,” and name the SQL dialect.
  • Few-shot Q→SQL examples teach your conventions (how “revenue” is defined) better than prose.
  • With many tables, retrieve the relevant few (schema RAG) instead of dumping everything — it’s Track 3 applied to metadata.
  • Sample rows prevent value mistakes ('Canada' vs 'CA', 'paid' vs 1).
Practice

Your database has a products(id, name, category, price) table and an order_items(order_id, product_id, qty) table. Write the schema-context lines you’d add to the prompt (with a description and one sample row each), and one few-shot example pair for “top 3 products by units sold.”

▸ Show solution & reasoning 
Table products(id, name, category, price REAL)
  -- one row per product. price in USD.
Table order_items(order_id, product_id, qty INTEGER)
  -- one row per product line in an order. product_id joins products.id

Sample: products    -> (7, 'USB-C Cable', 'Accessories', 9.99)
Sample: order_items -> (55, 7, 3)

Q: Top 3 products by units sold.
A: SELECT p.name, SUM(i.qty) AS units
   FROM order_items i JOIN products p ON p.id = i.product_id
   GROUP BY p.id
   ORDER BY units DESC
   LIMIT 3;

Why these choices: the descriptions state the grain (“one row per product line”) and the join key, which is what the model needs to avoid a wrong join. The sample rows pin down formats. The example demonstrates the exact pattern the question needs — a join, a SUM(qty) aggregate, group-by-then-order-and-limit — so a similar question (“top 5 categories by revenue”) now has a template to imitate. That’s few-shot teaching your house style, which is far more reliable than hoping the model infers it.

Your warehouse has 300 tables. Why not just put all their schemas in the prompt so the model “has everything it needs”?

  • A) It won’t fit the context window, costs far more tokens, and irrelevant tables actively distract the model — retrieve the relevant few instead.
    Correct. 300 schemas blow past the context window and the budget, and even if they fit, drowning the model in mostly-irrelevant tables lowers accuracy (it may join the wrong ones). Embedding table descriptions and retrieving only the handful relevant to the question keeps context small, cheap, and focused — schema RAG.
  • B) You can’t describe more than 10 tables to a model.
    There’s no hard limit like 10; the constraint is the context window and quality, not an arbitrary table count. The issue is that all 300 is both too big and too noisy — not that a bigger-than-tiny schema is impossible.
  • C) Models can only write SQL for one table at a time.
    Models happily write multi-table joins. Providing several relevant tables is expected; the point is to provide the relevant ones, not all 300.
  • D) Dumping all schemas is fine and always the best approach.
    It isn’t — it’s the classic mistake. It’s expensive, often impossible to fit, and hurts accuracy through distraction. Selective retrieval is the scalable pattern.

Why put the real tables and columns into the prompt?

  • So the model writes SQL against columns that actually exist instead of inventing plausible names.
    Correct. Schema grounding is what makes the generated query runnable and correct.
  • To make the prompt longer.
    Length isn't the goal; grounding is.
  • To raise the temperature.
    Unrelated to schema grounding.
  • To avoid needing a database.
    You still run against the database.
Deep dive: the schema you show is a design choice, not a given

It’s tempting to treat the schema as fixed input, but the best text-to-SQL engineers curate what the model sees. Raw production tables are often hostile to a language model: cryptic names (dim_cust_x), columns whose meaning depends on tribal knowledge, statuses encoded as integers, and metrics that require three joins and a filter to compute correctly. Pointing a model at that is asking for confident mistakes.

Instead, teams build a friendly surface just for the model — well-named views, a data dictionary of column meanings, and canonical definitions of business metrics — so that “revenue” is a real column on a view, not a fragile expression the model must reinvent every time.

This reframes the whole problem in your favor. The accuracy of a text-to-SQL system is determined less by the model and more by the quality of the semantic layer you put in front of it — the naming, the documentation, the example queries, the curated views. That is data modeling and metadata engineering, done for a new kind of consumer. When you present text-to-SQL work this way in an interview — “I’d invest in a clean semantic layer and a library of example queries, because that’s what moves accuracy” — you sound like someone who has actually shipped it, because that insight only comes from taking correctness seriously.

Now the model can write good SQL. Before we ever run it, we have to make sure it can’t hurt us. Next: generating safe, correct SQL.
