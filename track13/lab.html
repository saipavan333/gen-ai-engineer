Lab: Build a safe text-to-SQL runner

This is the one that plays to your decade of data work. You’ll spin up a real SQLite database in this page, write the safety layer that stands between a language model and your data, then try to break it — and watch it hold.

How to use this lab

Run the cells in order — they share memory. This is genuine SQLite (Python’s sqlite3 runs inside the browser), so the queries and the blocked attacks are all real.

Step 1 — Stand up a real database

An in-memory SQLite database with the two-table schema from this track. Nothing is installed and nothing leaves your machine.

step1_db.py
import sqlite3

con = sqlite3.connect(":memory:")
cur = con.cursor()
cur.executescript("""
CREATE TABLE customers (id INTEGER PRIMARY KEY, name TEXT, country TEXT);
CREATE TABLE orders (id INTEGER PRIMARY KEY, customer_id INTEGER, amount REAL, status TEXT);
INSERT INTO customers VALUES (1,'Ada Lovelace','Canada'),(2,'Bo Zhang','USA'),(3,'Cyrus Vane','Canada');
INSERT INTO orders VALUES (1,1,120.0,'paid'),(2,2,80.0,'paid'),(3,1,50.0,'refunded'),(4,3,200.0,'paid');
""")
con.commit()

print("tables:", [r[0] for r in cur.execute(
    "SELECT name FROM sqlite_master WHERE type='table'")])
print("orders loaded:", cur.execute("SELECT COUNT(*) FROM orders").fetchone()[0])

Step 2 — Write the guard

Never run generated SQL unchecked. Two rules stop the overwhelming majority of damage: one statement only (no semicolon-stacked payloads) and reads only. Build it, then test it against three candidates before it ever touches the database.

step2_guard.py
def validate(sql):
    q = sql.strip().rstrip(";").strip()
    if ";" in q:
        raise ValueError("only a single statement is allowed")
    if not q.lower().startswith(("select", "with")):
        raise ValueError("only SELECT queries are allowed")
    return q

for candidate in ["SELECT * FROM orders",
                  "DROP TABLE customers",
                  "SELECT 1; DROP TABLE customers"]:
    try:
        validate(candidate)
        print("ALLOWED :", candidate)
    except ValueError as e:
        print("BLOCKED :", candidate, "->", e)

Step 3 — Answer real questions

Now the useful half: the SQL a text-to-SQL model would generate for three business questions, run through the guard and executed for real.

step3_answer.py
def run_safe(sql, limit=100):
    q = validate(sql)
    rows = cur.execute(q + " LIMIT " + str(limit)).fetchall()
    cols = [d[0] for d in cur.description]
    return cols, rows

questions = {
  "Total revenue from paid orders":
      "SELECT SUM(amount) AS revenue FROM orders WHERE status = 'paid'",
  "Spend by customers in Canada":
      "SELECT SUM(o.amount) AS spent FROM orders o "
      "JOIN customers c ON c.id = o.customer_id WHERE c.country = 'Canada'",
  "Top customer by spend":
      "SELECT c.name, SUM(o.amount) AS spend FROM orders o "
      "JOIN customers c ON c.id = o.customer_id GROUP BY c.id ORDER BY spend DESC",
}

for question, sql in questions.items():
    cols, rows = run_safe(sql)
    print(question)
    print("   ", cols, "->", rows)

Step 4 — Try to break it

The real test. Throw the attacks a prompt-injected model might produce and confirm your data survives.

step4_attack.py
attacks = [
    "DROP TABLE customers",
    "SELECT * FROM orders; DELETE FROM orders",
    "UPDATE orders SET amount = 0",
]

for a in attacks:
    try:
        run_safe(a)
        print("!! GOT THROUGH:", a)
    except ValueError as e:
        print("blocked:", a, "->", e)

print()
print("customers still present:", cur.execute(
    "SELECT COUNT(*) FROM customers").fetchone()[0])
print("orders still present:   ", cur.execute(
    "SELECT COUNT(*) FROM orders").fetchone()[0])
The point of this lab

Every attack was blocked in application code — and in production you’d add the control that can’t be argued with at all: a read-only database user, so even a bug in this validator can’t destroy data. Defence in depth means assuming your own guard is imperfect.

Make it yours

1. Delete the semicolon check in validate and re-run Step 4. Which attack now gets through, and what does that tell you about stacked statements?
2. The run_safe helper appends LIMIT blindly. Find the query where that would break, and fix it.
3. Add a question of your own (“how many refunded orders?”), write the SQL, and run it.

▸ Show reasoning

1. SELECT * FROM orders; DELETE FROM orders gets through the prefix check because it starts with SELECT — and the second statement deletes your data. That’s why “starts with SELECT” alone is not a guard; stacked statements must be rejected outright.

2. Any query that already ends in a LIMIT (or uses one inside a subquery at the end) breaks with a syntax error when you append another. A robust runner parses the query or wraps it — e.g. SELECT * FROM ( … ) LIMIT 100 — rather than string-concatenating. Small detail, real production bug.

What you built
  • A real database running locally, seeded and queried.
  • A validator enforcing single-statement, read-only SQL.
  • A safe execution path with bounded results.
  • A live attack test proving the guard holds.
