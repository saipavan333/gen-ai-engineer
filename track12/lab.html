Lab: Advanced retrieval, measured

Basic retrieval fails on a question you’ll meet constantly: the user’s words don’t match the document’s. You’ll watch it fail, fix it with query expansion and rank fusion, add a reranker — and then prove the improvement with recall@k instead of claiming it.

How to use this lab

Run the cells in order — they share memory. Plain Python, running here.

Step 1 — Watch baseline retrieval miss

The user says “locked out”; the document says “account recovery”. Same meaning, different words — and a lexical scorer struggles.

step1_baseline.py
import math, re
from collections import Counter

DOCS = [
 "Refunds are issued within 5 business days to your original payment method.",
 "You can return any item within 30 days of delivery for a full refund.",
 "Standard shipping is free on orders over $50.",
 "To reset your password, click Forgot Password on the login page.",
 "If you are locked out of your account, use the account recovery form.",
 "Gift cards are non-refundable and never expire.",
]
STOP = {"the","a","an","is","are","to","of","for","on","in","do","my","you","your","can","how","and","it","if"}

def vec(t):
    return Counter(w for w in re.findall(r"[a-z0-9]+", t.lower())
                   if len(w) > 2 and w not in STOP)

def cos(a, b):
    sh = set(a) & set(b)
    dot = sum(a[w] * b[w] for w in sh)
    na = math.sqrt(sum(v*v for v in a.values()))
    nb = math.sqrt(sum(v*v for v in b.values()))
    return dot / (na * nb) if na and nb else 0.0

INDEX = [(d, vec(d)) for d in DOCS]

def search(q, k=3):
    qv = vec(q)
    s = [(cos(qv, dv), d) for d, dv in INDEX]
    s.sort(reverse=True, key=lambda x: x[0])
    return s[:k]

QUERY = "I cannot sign in and need to get back into my profile"
print("baseline for:", QUERY)
for sc, d in search(QUERY):
    print("  ", round(sc, 3), "|", d[:54])

Step 2 — Expand the query, then fuse the rankings

Ask the same thing several ways, retrieve for each, and reward documents that rank well across all of them. That’s reciprocal rank fusion.

step2_expand.py
EXPANSIONS = {QUERY: ["reset password", "account recovery", "locked out of account"]}

def rrf(queries, k=3, kconst=60):
    scores = {}
    for q in queries:
        for rank, (sc, d) in enumerate(search(q, k=len(DOCS))):
            if sc <= 0:
                continue
            scores[d] = scores.get(d, 0) + 1 / (kconst + rank)
    return sorted(scores.items(), key=lambda x: -x[1])[:k]

queries = [QUERY] + EXPANSIONS[QUERY]
print("expanded + fused:")
for d, s in rrf(queries):
    print("  ", round(s, 4), "|", d[:54])

Step 3 — Rerank the shortlist

Retrieval is cheap and approximate; reranking is expensive and accurate, so you run it only on the shortlist. Here “coverage” stands in for a cross-encoder.

step3_rerank.py
def rerank(query, candidates):
    qw = set(vec(query)) | set(vec(" ".join(EXPANSIONS.get(query, []))))
    out = []
    for d in candidates:
        dw = set(vec(d))
        coverage = len(qw & dw) / max(1, len(qw))
        out.append((round(coverage, 3), d))
    out.sort(reverse=True, key=lambda x: x[0])
    return out

shortlist = [d for d, _ in rrf(queries, k=4)]
for sc, d in rerank(QUERY, shortlist):
    print("  ", sc, "|", d[:54])

Step 4 — Prove it with recall@k

Never claim an improvement you haven’t measured. Define the right answer for each question, then score both pipelines.

step4_measure.py
GOLD = {
  QUERY: "If you are locked out of your account, use the account recovery form.",
  "How long do refunds take?": "Refunds are issued within 5 business days to your original payment method.",
}

def recall_at_k(fn, k=2):
    hits = 0
    for q, gold in GOLD.items():
        got = fn(q, k)
        if gold in got:
            hits += 1
    return hits / len(GOLD)

def pipeline_baseline(q, k):
    return [d for _, d in search(q, k)]

def pipeline_advanced(q, k):
    return [d for d, _ in rrf([q] + EXPANSIONS.get(q, []), k)]

print("recall@2 baseline:", recall_at_k(pipeline_baseline))
print("recall@2 advanced:", recall_at_k(pipeline_advanced))
Make it yours

1. Re-run recall_at_k with k=3 instead of k=5. Does fusion still beat the baseline, and why does k change the story?
2. Reranking added real latency. How would you decide whether to keep it?

▸ Show reasoning

1. A smaller k is a harder test, so the gap usually widens. At large k the right document is somewhere in the pile and everything looks healthy — which is exactly how weak retrieval hides. Report recall at the k you actually feed the model, not the k that flatters you.

2. Measure the recall lift against the added p95 latency and cost per query. If reranking buys two points of recall and doubles latency, drop it — or apply it selectively, only when the top result's score falls below a confidence threshold. "Always on" is a decision too, and it should be one you measured.

What you built
  • A reproduction of the vocabulary-mismatch failure — the one that breaks real RAG.
  • Query expansion + reciprocal rank fusion.
  • A rerank stage over a cheap shortlist.
  • recall@k measured before and after — evidence, not opinion.

“I improved retrieval” means nothing. “I lifted recall@2 from 0.5 to 1.0 on our golden set with query expansion” is an engineering claim.
