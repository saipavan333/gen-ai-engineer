Track 6 · Interview check

Interview check: Evals & Observability

This is the topic that most reveals seniority. Anyone can build a demo; only people who measure can ship and improve. Interviewers probe hard here — they want to hear a measurement mindset, not prompt-tinkering.

The drill

Answer aloud first. The throughline: you can’t improve what you can’t measure; gate changes offline, learn online, see everything via tracing, and let the golden set grow forever.

The questions teams ask

Q1 · Framing

“Why do evals matter, and what does a basic one need?”

▸ Strong answer

Because LLM outputs are open-ended and non-deterministic, so without measurement every change is a guess and regressions go unnoticed. A basic eval needs three things: a golden set (representative inputs with known-good answers/criteria), a metric (exact match, keywords, or LLM-as-judge), and a baseline to compare against. The loop is change → score → compare → keep or revert. Evals are what turn “feels better” into “is better.”

Q2 · Golden sets

“How do you build a good evaluation set?”

▸ Strong answer

Mine real inputs (not invented happy-path), label correct answers or acceptance criteria, deliberately include edge cases and known failures, and cover all categories so a fix to one can’t silently break another. Keep it held out (don’t overfit) and living (every production failure becomes a permanent case). Small and representative (30–50 sharp cases) beats large and random. Score per category to localize regressions.

Q3 · LLM-as-judge

“How do you evaluate open-ended outputs, and what are the risks?”

▸ Strong answer

Use LLM-as-judge: prompt a capable model with a clear rubric (and a reference when available) to score the output, forcing a structured verdict you can aggregate. Pairwise (“which is better?”) is often more reliable than absolute scores. Risks: verbosity, position, and self-preference biases, and leniency. Mitigate with a rubric that says what to ignore, swapping positions, using a different model family as judge, and — essential — validating the judge against human labels before trusting it.

Q4 · Offline vs online

“Difference between offline and online evaluation?”

▸ Strong answer

Offline runs on a fixed golden set before deploy — fast, cheap, repeatable, great for gating changes, but it’s a proxy. Online measures real production traffic — A/B tests, user feedback, outcome metrics — the ground truth, but slower and riskier. You need both: offline to ship safely, online to learn what actually works. Standard flow is offline gate → small online rollout → full rollout, never straight to 100%, and always watch guardrail metrics (latency, cost, errors).

Q5 · Observability

“A RAG bot gives a wrong answer in production. How does tracing help?”

▸ Strong answer

A trace records each step (span) of the request with inputs/outputs/timing/cost. I open that request’s trace and check the retrieve span: if the relevant chunk wasn’t fetched, it’s a retrieval problem; if it was fetched but the answer ignored it, it’s a generation/grounding problem. Logging the exact assembled prompt and retrieved context is what makes the failure reproducible. Tracing turns “it’s broken” into a precise, single-request diagnosis — and it’s a privacy-sensitive log, so redact PII.

Q6 · Regression testing

“How do you stop prompt changes from silently breaking things?”

▸ Strong answer

Treat prompts like code: version them, and run the eval suite as a CI gate on every change. The gate blocks merges that regress critical cases or drop the overall score. Keep tests non-flaky with temperature 0 and structural/keyword or judge-threshold assertions instead of exact strings. Every production incident becomes a new permanent test, so the same bug can’t ship twice. It’s unit tests + CI applied to LLM behavior.

Q7 · Applied

“How would you evaluate a RAG system end to end?”

▸ Strong answer

Measure the two stages separately (Track 3.8): retrieval with recall@k / MRR on a golden set of (query → relevant chunk) labels, and generation with faithfulness and answer correctness via LLM-as-judge. Run it offline as a CI gate, trace production to mine new failures into the golden set, and A/B test changes online watching answer quality plus latency/cost guardrails. Separating retrieval from generation tells me which half to fix — the core RAG debugging skill.

Rapid-fire self-check

The single most defensible asset in an applied-GenAI team is usually…

  • A) A well-built, growing evaluation set that captures what “good” means for the task.
    Correct. Prompts are easy to copy and models are a commodity, but a sharp eval set encoding your task’s definition of quality, edge cases, and failures is hard-won and lets you adopt new models, refactor safely, and prove improvement. It’s the real moat and the engine of fast iteration.
  • B) The exact wording of the system prompt.
    Prompts are trivial to copy and frequently rewritten; they’re not defensible. What lets you improve prompts safely is the eval set behind them.
  • C) Access to the largest model.
    Models are widely available and change constantly; access isn’t a durable advantage. The ability to measure which model/config is best for your task is.
  • D) A big GPU cluster.
    Hardware is rentable and not task-specific. The differentiator is knowing what to optimize for — which your evaluation defines.

Your offline eval improves but you still A/B test before full rollout. Why bother, if offline already passed?

  • A) Offline is a proxy that can miss real-world query types, behaviors, and trade-offs; online confirms true impact and catches what the golden set didn’t.
    Correct. An offline win is necessary but not sufficient — the golden set can’t capture everything. A staged A/B validates on real traffic and surfaces regressions (including latency/cost) the proxy missed, before exposing all users.
  • B) Because offline evals are unreliable and shouldn’t be trusted.
    Offline evals are your essential gate — the point isn’t distrust, it’s that they’re a proxy to be confirmed online. Both stages matter.
  • C) To slow down releases on principle.
    The goal is safety and learning, not slowness. A/B testing is about validating impact and catching surprises, not adding friction for its own sake.
  • D) A/B tests replace the need for offline evals.
    They’re complementary: offline gates cheaply before exposure; online validates truthfully after. You use both in sequence, not one instead of the other.

  • Building a golden set (real inputs, edge cases, living, per-category).
  • LLM-as-judge: rubric, pairwise, biases, validate against humans.
  • Offline vs online; the staged rollout; guardrail metrics.
  • Tracing/spans for debugging, and PII in logs.
  • Regression testing as CI; keeping LLM tests non-flaky.
  • Evaluating RAG (retrieval vs generation) and agents end to end.
