Lab: Build an eval harness that catches a regression

This is the skill that separates people who ship AI from people who demo it. You’ll build a golden set, a scorer, and a quality gate — then watch a “friendlier” new version get blocked because your harness caught the damage it did.

How to use this lab

Run the cells in order — they share memory. Everything is plain Python and runs in this page.

Step 1 — Write a golden set

A golden set is a fixed list of cases with known-good expectations. Note the last case: the right answer is a refusal. Testing that your system knows when to say “I don’t know” is as important as testing what it knows.

step1_golden.py
GOLDEN = [
    {"q": "How long do refunds take?",      "must_have": ["5", "business days"]},
    {"q": "Is shipping free?",              "must_have": ["free", "50"]},
    {"q": "How do I reset my password?",    "must_have": ["forgot password"]},
    {"q": "Do gift cards expire?",          "must_have": ["never expire"]},
    {"q": "What is the capital of France?", "must_have": ["don't know"]},
]

print("golden set size:", len(GOLDEN))
for row in GOLDEN:
    print(" -", row["q"], "->", row["must_have"])

Step 2 — Score an answer

Scoring must be automatic or it won’t happen. This one checks that the required facts appear; production harnesses add exact-match for structured fields and an LLM judge for open-ended prose.

step2_score.py
def score_answer(answer, must_have):
    a = answer.lower()
    hits = [k for k in must_have if k.lower() in a]
    return len(hits) / len(must_have), hits

print(score_answer("Refunds take 5 business days.", ["5", "business days"]))
print(score_answer("Usually about a week.",        ["5", "business days"]))

Step 3 — Evaluate two versions

Here are two “models”. v2 is the kind of change that feels like an improvement — warmer, more helpful, less likely to refuse. Run it and read the per-case scores carefully.

step3_evaluate.py
def model_v1(q):
    ql = q.lower()
    if "refund" in ql:   return "Refunds take 5 business days to your original payment method."
    if "shipping" in ql: return "Shipping is free on orders over $50."
    if "password" in ql: return "Click Forgot Password on the login page."
    if "gift" in ql:     return "Gift cards never expire."
    return "I don't know based on the documents."

def model_v2(q):   # a 'friendlier' rewrite that stopped refusing
    ql = q.lower()
    if "refund" in ql:   return "Refunds are processed quickly, usually within a week."
    if "shipping" in ql: return "Shipping is free on orders over $50."
    if "password" in ql: return "Click Forgot Password on the login page."
    if "gift" in ql:     return "Gift cards never expire."
    return "The capital of France is Paris."

def evaluate(model):
    total, rows = 0.0, []
    for case in GOLDEN:
        ans = model(case["q"])
        s, _ = score_answer(ans, case["must_have"])
        total += s
        rows.append((round(s, 2), case["q"][:30], ans[:44]))
    return total / len(GOLDEN), rows

for name, m in [("v1", model_v1), ("v2", model_v2)]:
    avg, rows = evaluate(m)
    print("\n===", name, " average score:", round(avg, 3))
    for r in rows:
        print("   ", r)

Step 4 — Gate the deploy

The number only matters if it can stop something. This is the whole mechanism: score the change, compare to a threshold, exit non-zero on failure — which is what turns a red build in CI.

step4_gate.py
THRESHOLD = 0.9

def gate(name, model):
    avg, _ = evaluate(model)
    ok = avg >= THRESHOLD
    print(name, "scored", round(avg, 2), "| threshold", THRESHOLD, "->",
          "PASS - ship it" if ok else "BLOCKED - do not deploy")
    return ok

gate("v1", model_v1)
gate("v2", model_v2)
print("\nv2 sounded friendlier — and silently broke two cases.")
print("Without this gate, it ships. With it, the build goes red.")
What just happened

v2 lost the refund fact and stopped refusing the unanswerable question — it confidently claimed Paris. Both are exactly the kind of regression no human eyeballing three outputs would catch. Your harness caught both in under a second.

Make it yours

1. Add a case whose right answer is a refusal (something the docs don’t cover). Does v1 still pass?
2. Lower THRESHOLD to 0.7 — v2 now passes. What does that teach you about picking a threshold?
3. A real production failure arrives: a user asked something and got a bad answer. Add it to GOLDEN. You’ve just made that bug impossible to reintroduce.

▸ Show reasoning

2. A threshold set to whatever makes the current build pass is theatre. Pick it from what the product actually requires, and only lower it deliberately with a reason. A gate you relax whenever it's inconvenient stops being a gate.

3. This is the flywheel: every real failure becomes a permanent test case, so the system can never regress the same way twice. Over months that golden set becomes the most valuable asset you own — competitors can copy your prompts, not your library of hard-won failures.

What you built
  • A golden set including a refusal case.
  • An automatic scorer.
  • An A/B evaluation across two versions.
  • A quality gate that blocks a regression — the core of eval-driven development.
