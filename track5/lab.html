Lab: Prepare data and size a LoRA

Fine-tuning is 90% data work and a little arithmetic. You’ll build a training dataset, run the quality checks that decide whether the tune succeeds, compute exactly how few parameters LoRA trains — and write the decision function that tells you whether to fine-tune at all.

How to use this lab

Run the cells in order — they share memory. No GPU required: this is the preparation and the maths, which is where the real decisions live.

Step 1 — Build the dataset

Instruction tuning wants a consistent shape: an instruction, an optional input, and the output you want the model to imitate.

step1_dataset.py
import json

RAW = [
    {"question": "How long do refunds take?",   "answer": "Refunds take 5 business days."},
    {"question": "Is shipping free?",           "answer": "Shipping is free on orders over $50."},
    {"question": "How do I reset my password?", "answer": "Use Forgot Password on the login page."},
]

def to_instruction(row):
    return {"instruction": "Answer the customer's question in one sentence.",
            "input": row["question"], "output": row["answer"]}

dataset = [to_instruction(r) for r in RAW]
for d in dataset[:2]:
    print(json.dumps(d))
print("examples:", len(dataset))

Step 2 — Quality checks that decide the outcome

Duplicates teach the model to over-weight one answer; short outputs teach it to be unhelpful. Catching these beats any hyper-parameter tuning.

step2_quality.py
def check(rows, min_len=10):
    clean, problems, seen = [], [], set()
    for i, d in enumerate(rows):
        if not d.get("output") or len(d["output"]) < min_len:
            problems.append((i, "output too short")); continue
        key = d["input"].strip().lower()
        if key in seen:
            problems.append((i, "duplicate input")); continue
        seen.add(key); clean.append(d)
    return clean, problems

noisy = dataset + [dataset[0], {"instruction": "x", "input": "hi", "output": "ok"}]
clean, problems = check(noisy)
print("kept:", len(clean), "of", len(noisy))
print("dropped:", problems)

Step 3 — Why LoRA is cheap (the actual numbers)

Full fine-tuning updates every weight. LoRA freezes them and trains two small matrices per projection. Run this and see the ratio for yourself — it’s the reason you can tune on free Colab.

step3_lora_math.py
def full_params(d_model, n_layers):
    return 4 * d_model * d_model * n_layers        # 4 big d x d projections per layer

def lora_params(d_model, n_layers, rank):
    return 4 * 2 * d_model * rank * n_layers     # two d x r matrices instead

d, layers = 4096, 32
full = full_params(d, layers)
print("full fine-tune trainable params:", format(full, ","))
for r in (4, 8, 16, 64):
    lo = lora_params(d, layers, r)
    print("  rank", r, "-> ", format(lo, ","), " = ",
          round(lo / full * 100, 3), "% of the weights")
Read that output again

At rank 8 you train well under 1% of the parameters — which is why the adapter is a few megabytes, why it fits in free GPU memory, and why you can keep one base model with many swappable adapters.

Step 4 — Should you fine-tune at all?

The most valuable output of this track is often “don’t”. Encode the decision so it isn’t a vibe.

step4_decide.py
def should_finetune(need_fresh_facts, need_style, examples, tried_prompting):
    if need_fresh_facts:
        return "No - use RAG. Fine-tuning does not add changing knowledge."
    if not tried_prompting:
        return "Not yet - try prompting and few-shot first."
    if examples < 200:
        return "Probably not - gather more examples first."
    if need_style:
        return "Yes - consistent style/format is what fine-tuning is for."
    return "Maybe - measure whether prompting already clears the bar."

print(should_finetune(True,  False, 5000, True))
print(should_finetune(False, True,  1000, True))
print(should_finetune(False, True,  50,   True))
print(should_finetune(False, True,  1000, False))
Make it yours

1. Change the rank from 8 to 64 and recompute the trainable percentage. At what point does LoRA stop being the cheap option?
2. You have 200 clean examples and a factual-accuracy problem. Run should_finetune() — then defend the answer in your own words.

▸ Show reasoning

1. Trainable parameters grow roughly linearly with rank, so rank 64 is about eight times rank 8. It's still far below full fine-tuning, but the gap narrows while the quality gain flattens — most tasks see little benefit above rank 16–32. High rank means you're paying for capacity you can't fill with a small dataset.

2. Don't fine-tune. Two hundred examples is thin, and more importantly fine-tuning teaches form — tone, structure, output format — not facts. A factual-accuracy problem is a retrieval problem. Fine-tuning on it produces a model that's confidently wrong in a more consistent voice.
