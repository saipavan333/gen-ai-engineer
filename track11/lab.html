Step 1 — Describe your project

Note what this structure forces you to have: decisions with reasons, measured results, and honest limitations. If you can’t fill these in, the project isn’t finished — and that’s useful to learn now rather than in an interview.

step1_project.py
PROJECT = {
    "name": "docs-assistant",
    "one_liner": "A grounded Q&A assistant over a company's help-center docs.",
    "stack": ["Python", "Chroma", "Ollama (llama3.2)", "sentence-transformers"],
    "decisions": [
        ("Chunk size 500 with 50 overlap",
         "200-char chunks split definitions across boundaries and hurt 'what is X' answers"),
        ("Hybrid BM25 + dense retrieval",
         "pure vector search missed exact product names"),
    ],
    "results": {"golden_set": 40, "recall_at_5": 0.86, "faithfulness": 0.92},
    "limitations": ["multi-hop questions need query decomposition",
                    "no reranker yet"],
    "run": ["pip install -r requirements.txt", "ollama pull llama3.2",
            "python ingest.py", "python app.py"],
}
print(PROJECT["name"], "-", PROJECT["one_liner"])

Step 2 — Generate the README

step2_render.py
def render(p):
    L = []
    L.append("# " + p["name"] + " - " + p["one_liner"])
    L.append("")
    L.append("## Stack")
    L.append(", ".join(p["stack"]))
    L.append("")
    L.append("## Key decisions and trade-offs")
    for choice, why in p["decisions"]:
        L.append("- **" + choice + "** - " + why + ".")
    L.append("")
    L.append("## Results (measured)")
    r = p["results"]
    L.append("On a " + str(r["golden_set"]) + "-question golden set: recall@5 = "
             + str(r["recall_at_5"]) + ", faithfulness = " + str(r["faithfulness"]) + ".")
    L.append("")
    L.append("## Limitations and next steps")
    for lim in p["limitations"]:
        L.append("- " + lim)
    L.append("")
    L.append("## Run it")
    for c in p["run"]:
        L.append("    " + c)
    return "\n".join(L)

readme = render(PROJECT)
print(readme)

Step 3 — Score it against the hiring checklist

A reviewer spends about ninety seconds. These five things decide whether they keep reading.

step3_score.py
CHECKS = {
    "one-line description up top": lambda t: t.startswith("# ") and " - " in t.splitlines()[0],
    "explains decisions":          lambda t: "Key decisions" in t,
    "reports measured results":    lambda t: "Results (measured)" in t,
    "states limitations":           lambda t: "Limitations" in t,
    "has run instructions":        lambda t: "Run it" in t,
}

def score_readme(text):
    passed = {name: fn(text) for name, fn in CHECKS.items()}
    return sum(passed.values()) / len(passed), passed

score, detail = score_readme(readme)
for name, ok in detail.items():
    print(("PASS" if ok else "MISS"), "|", name)
print("score:", round(score * 100), "%")

Step 4 — Compare with a typical README

step4_compare.py
WEAK = "# docs-assistant\n\nA RAG app I built.\n\n## Setup\nRun main.py\n"

weak_score, weak_detail = score_readme(WEAK)
print("typical README score:", round(weak_score * 100), "%")
for name, ok in weak_detail.items():
    if not ok:
        print("   missing:", name)

print("\nSame project. The difference is evidence.")
Make it yours

1. Paste the README of a project you've actually shipped into score_readme. Which check fails first?
2. You have no measured results because you never built a golden set. What's the fastest honest fix — and what do you write if the numbers are mediocre?

▸ Show reasoning

1. Almost always "reports measured results" and "states limitations". Most READMEs describe what was built and how to install it — the two sections that signal engineering judgment are the ones people skip.

2. Write twenty questions you expect the system to handle, label the correct answer or source, and score against them. That's an afternoon. And publish the mediocre number: "recall@5 = 0.71, and here's what I'd fix next" reads as an engineer who measures. A README with no numbers reads as someone who never checked.
