Lab: Catch bias with disaggregated evaluation

A screening model reports 83% accuracy and everyone celebrates. You’ll build the evaluation that looks underneath that number — and find the group it quietly fails, using matched candidates whose only difference is their name.

How to use this lab

Run the cells in order — they share memory. This is the exact technique you’d use before letting any model touch decisions about people.

Step 1 — Build matched pairs

The trick is control: identical qualifications, only the name differs. Any difference in outcome can then only come from the name.

step1_cases.py
# (group, candidate, correct decision) — qualifications are matched across groups
CASES = [
    ("group_a", "Ada Lovelace, 8 years experience, Python", "advance"),
    ("group_a", "Grace Hopper, 10 years experience, C++",  "advance"),
    ("group_a", "Alan Turing, 6 years experience, ML",     "advance"),
    ("group_b", "Amara Okafor, 8 years experience, Python", "advance"),
    ("group_b", "Wei Zhang, 10 years experience, C++",      "advance"),
    ("group_b", "Priya Nair, 6 years experience, ML",       "advance"),
]
print("matched cases:", len(CASES))
print("every candidate here is qualified - the correct answer is always 'advance'")

Step 2 — Two screeners

v1 judges on years of experience. v2 also picked up a shortcut from its training data — the kind nobody writes on purpose and nobody notices without this test.

step2_models.py
def years_of(text):
    return int(text.split(",")[1].strip().split()[0])

def screener_v1(text):                       # judges qualifications only
    return "advance" if years_of(text) >= 5 else "reject"

SHORTCUT = ("amara", "wei", "priya")       # names it saw less often in training

def screener_v2(text):                       # accidentally learned a name shortcut
    name = text.split(",")[0].lower()
    if any(h in name for h in SHORTCUT) and years_of(text) < 9:
        return "reject"
    return "advance" if years_of(text) >= 5 else "reject"

print(screener_v2("Alan Turing, 6 years experience, ML"))
print(screener_v2("Priya Nair, 6 years experience, ML"))

Step 3 — The number that hides the problem

Now measure overall accuracy, then split it by group. This one contrast is the whole lesson.

step3_disaggregate.py
def accuracy(model, cases):
    return sum(1 for g, t, y in cases if model(t) == y) / len(cases)

def by_group(model, cases):
    groups = {}
    for g, t, y in cases:
        groups.setdefault(g, [0, 0])
        groups[g][1] += 1
        if model(t) == y:
            groups[g][0] += 1
    return {g: round(c / n, 2) for g, (c, n) in groups.items()}

for name, m in [("v1", screener_v1), ("v2", screener_v2)]:
    print(name, "| overall accuracy:", round(accuracy(m, CASES), 2),
          "| by group:", by_group(m, CASES))
Look at v2 carefully

Its overall score still looks respectable — but the per-group split shows one group scoring far below the other, on identical qualifications. Ship on the aggregate and you ship discrimination you never measured.

Step 4 — Gate on the gap, not the average

Make the disparity a first-class metric with its own threshold, exactly like a quality gate.

step4_gate.py
def disparity(model, cases):
    g = by_group(model, cases)
    return round(max(g.values()) - min(g.values()), 2)

MAX_GAP = 0.05

for name, m in [("v1", screener_v1), ("v2", screener_v2)]:
    gap = disparity(m, CASES)
    print(name, "| disparity", gap, "->",
          "PASS" if gap <= MAX_GAP else "BLOCKED - do not ship")
Make it yours

1. Remove the name from the input before screening. Does the gap close — and is “just delete the name” a complete fix in the real world?
2. The gap can’t be closed and the launch date is next week. What do you recommend, and to whom?

▸ Show reasoning

1. It closes here because the name was the only signal. In reality bias hides in proxies — postcode, school, employment gaps, phrasing — so deleting the obvious field rarely suffices. You have to keep measuring outcomes per group, not just remove the field you thought was responsible.

2. Recommend not automating the decision: use the model to assist reviewers rather than reject candidates, document the known disparity, and put it in front of whoever owns the legal and reputational risk. Under risk-based regulation this is a high-risk use, and "we shipped knowing it was unfair" is the worst possible position — technically, ethically, and legally.

What you built
  • Matched-pair test cases that isolate a single variable.
  • Disaggregated accuracy — the measurement the aggregate hides.
  • A disparity metric with a hard gate.
  • The judgment to decline to automate when the gap won't close.

You can now measure fairness, not just accuracy — the skill that separates responsible engineers from confident ones.
