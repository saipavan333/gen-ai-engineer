Lab: Document extraction you can trust

Turning a scanned invoice into database rows is the highest-value multimodal task in business — and the easiest to get dangerously wrong. You’ll extract the fields, validate them, cross-check the arithmetic, and build the routing rule that decides what a human must see.

How to use this lab

Run the cells in order — they share memory. We start from the text a vision model or OCR would hand you, which is where the engineering actually begins.

Step 1 — The document as text

This is what comes out of OCR or a VLM: roughly structured, inconsistently spaced, no schema.

step1_ocr.py
OCR_TEXT = """ACME SUPPLIES LTD
Invoice #  INV-2025-0473
Date: 14/03/2025

Description        Qty   Price
Widget A            3    40.00
Cable pack          2    15.50
Shipping                  9.99

TOTAL DUE   $ 160.99
Payment terms: net 30
"""

print(OCR_TEXT)

Step 2 — Extract the fields

A VLM can return JSON directly; either way you must pull structure out of text and never assume it succeeded.

step2_extract.py
import re

def extract(text):
    out = {}
    m = re.search(r"invoice\s*#?\s*([A-Z0-9\-]+)", text, re.I)
    out["invoice_no"] = m.group(1) if m else None
    m = re.search(r"date:\s*([0-9]{1,2}/[0-9]{1,2}/[0-9]{4})", text, re.I)
    out["date"] = m.group(1) if m else None
    m = re.search(r"total\s*due\s*\$?\s*([0-9]+(?:\.[0-9]{2})?)", text, re.I)
    out["total"] = float(m.group(1)) if m else None
    out["vendor"] = text.strip().splitlines()[0].strip()
    return out

fields = extract(OCR_TEXT)
print(fields)

Step 3 — Validate and normalise

Extraction gives you strings. Validation decides whether they’re usable — and normalising the date is where silent disasters hide (is 14/03 March or December?).

step3_validate.py
from datetime import datetime

def normalize_date(d):
    for fmt in ("%d/%m/%Y", "%m/%d/%Y"):
        try:
            return datetime.strptime(d, fmt).strftime("%Y-%m-%d")
        except ValueError:
            pass
    return None

def validate(f):
    errs = []
    if not f.get("invoice_no"): errs.append("missing invoice number")
    if not f.get("vendor"):     errs.append("missing vendor")
    if f.get("total") is None:
        errs.append("missing total")
    elif not (0 < f["total"] < 100000):
        errs.append("total out of plausible range")
    iso = normalize_date(f.get("date") or "")
    if not iso:
        errs.append("unparseable date")
    else:
        f["date"] = iso
    return f, errs

fields, errs = validate(fields)
print(fields)
print("problems:", errs or "none")

Step 4 — Cross-check, then decide who sees it

The strongest defence against a confident wrong number is a second, independent calculation. Then route by risk — because paying the wrong invoice is not reversible.

step4_route.py
LINE_ITEMS = [("Widget A", 3, 40.00), ("Cable pack", 2, 15.50), ("Shipping", 1, 9.99)]

def cross_check(total, items, tolerance=0.01):
    computed = round(sum(q * p for _, q, p in items), 2)
    return computed, abs(computed - total) <= tolerance

computed, matches = cross_check(fields["total"], LINE_ITEMS)
print("line items sum to", computed, "| invoice says", fields["total"], "| match:", matches)

def route(f, errs, matches, auto_limit=500):
    if errs:            return "HUMAN REVIEW - extraction problems: " + str(errs)
    if not matches:    return "HUMAN REVIEW - total does not match line items"
    if f["total"] > auto_limit:
        return "HUMAN APPROVAL - amount above the auto-pay limit"
    return "AUTO-APPROVED"

print(route(fields, errs, matches))
Make it yours

1. Change the invoice total to 1600.99 in OCR_TEXT and re-run Steps 2–4. Which guard catches it — and would a VLM alone have noticed?
2. The date 14/03/2025 is unambiguous, but 03/04/2025 is not. What should the system do when both formats parse?

▸ Show reasoning

1. The cross-check catches it — the line items sum to 160.99 and the stated total doesn't match, so it routes to a human. A VLM reading the document alone would happily report 1600.99 with full confidence, because it has nothing to compare against. Independent verification beats a better reader.

2. Don't guess. Either use a locale/vendor convention you can justify, or flag it for review. A silently mis-parsed date can shift a payment by months, and nothing in the output would look wrong.

What you built
  • Field extraction from realistic, messy document text.
  • Validation and normalisation, including the date trap.
  • An independent cross-check against line items.
  • Risk-based routing — auto-approve, review, or require a human.

Accuracy alone doesn't make extraction shippable. Calibrated uncertainty and a human handoff do.
