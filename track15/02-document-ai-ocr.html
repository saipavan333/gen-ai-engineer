Document AI & OCR

Turning documents into structured data — invoices, forms, receipts, contracts — is the highest-value multimodal use case in business, and one VLMs changed overnight. This lesson covers OCR, the new VLM-based approach, and how to make extraction trustworthy.

The problem: documents aren’t data

A scanned invoice is a picture, not a database row. To use it — to pay it, audit it, total it — someone has to turn “this image” into “vendor: Acme, total: $4,210, due: 2025-07-01.” That translation, done by hand, is a vast amount of human labor across finance, insurance, healthcare, and logistics. Document AI is the task of doing it automatically: from a document image, produce clean structured fields.

The classic first step is OCR (optical character recognition) — detecting and transcribing the text in an image. Open tools like Tesseract do this for free. OCR gives you the raw words and roughly where they are, but not their meaning: it returns a wall of text, not “this number is the total and that one is the tax.”

The old way vs the new way

The old way: OCR the document, then write rules and regular expressions to find fields (“the total is the number after the word ‘Total’”). This is brittle — every vendor’s invoice has a different layout, so the rules break constantly and each new format is a new engineering project. Teams spent years on this.

The new way: hand the document image straight to a VLM and ask for structured output. The model reads the text and understands the layout, so it can find the total whether it’s top-right, bottom, or in a table — no per-vendor rules. You combine two things you already know: multimodal input (Lesson 10.1) and structured JSON output (Track 2), ideally validated against a schema (Track 7).

extract_invoice.py
import ollama, json
from pydantic import BaseModel

class Invoice(BaseModel):        # the shape we require back
    vendor: str
    total: float
    due_date: str

prompt = """Read this invoice. Return ONLY JSON with keys:
vendor (string), total (number), due_date (YYYY-MM-DD). No prose."""

reply = ollama.chat(model="llava",
    messages=[{"role": "user", "content": prompt, "images": ["invoice.png"]}])

data = Invoice(**json.loads(reply["message"]["content"]))   # parse + validate
print(data.vendor, data.total, data.due_date)

The pydantic model does real work: if the VLM returns a total as "$4,210" instead of a number, or omits due_date, validation fails loudly instead of passing bad data downstream — the “assume it’ll misbehave” discipline from Start Here, applied to extraction.

Best of both: OCR + VLM

For accuracy-critical fields, combine them. OCR is often more reliable at exact character transcription (an account number, a precise amount), while the VLM is better at understanding which field is which and handling layout. A strong pattern is to use the VLM for structure and OCR to verify the exact digits — cross-checking the two, and flagging disagreements for human review. This is defense in depth again: two imperfect readers catch each other’s mistakes.

Key points
  • Document AI = image of a document → clean structured fields; it’s the highest-value multimodal use case.
  • OCR (e.g. free Tesseract) transcribes the text but not its meaning; rules on top are brittle across layouts.
  • VLMs read text and layout, extracting fields with no per-vendor rules — multimodal input + structured JSON output.
  • Validate the JSON against a schema (pydantic) so bad extractions fail loudly, not silently.
  • For critical fields, combine OCR + VLM and flag disagreements for a human — defense in depth.

Why it’s not enough: this is high-stakes — a wrong total or account_number sends real money to the wrong place, and a VLM can misread a digit or hallucinate a plausible value with full confidence and valid-looking JSON. Schema validation confirms the shape (a number, a string) but not the correctness of the value.

What I’d add: (1) cross-check exact fields with OCR and require agreement on account_number and total; (2) sanity rules (total within an expected range, account number matches the expected format/length); (3) a confidence/uncertainty signal, and (4) a human-in-the-loop confirmation step for any payment above a threshold or any case where the checks disagree. The principle: the amount of automation should scale with how reversible and low-cost a mistake is — and irreversible money movement demands verification and, above a threshold, a human. This is exactly the judgment (Track 7, Track 1) that separates a safe system from a demo.

Why do VLMs largely replace the old “OCR + hand-written regex rules” approach to extracting invoice fields?

  • A) VLMs understand layout and meaning, so they find the right field across many different document formats without per-format rules.
    Correct. The brittleness of the old approach came from rules tied to specific layouts (“total is after the word Total”), which broke on every new vendor. A VLM reads the document holistically and identifies fields regardless of position, eliminating most of that per-format engineering.
  • B) VLMs never make mistakes, so no validation is needed.
    VLMs absolutely make mistakes (misread digits, hallucinated values), which is why schema validation and, for critical fields, OCR cross-checks and human review remain essential. The advantage is flexibility across layouts, not infallibility.
  • C) VLMs don’t need to read the text at all.
    They very much read the text — that’s the point. The improvement over pure OCR is that they also understand which text is which field, using layout and context.
  • D) OCR can’t run for free, but VLMs can.
    OCR has excellent free options (Tesseract). Cost isn’t the reason; the reason is that VLMs handle varied layouts and meaning without brittle per-format rules.
Deep dive: how you evaluate an extraction system

Document extraction is a place where the evaluation discipline from Track 6 becomes very concrete, and interviewers love it because it’s measurable. You build a golden set of documents with hand-verified correct fields, then measure field-level accuracy — for each field (vendor, total, date), what fraction of documents did the system get exactly right?

Aggregate accuracy hides problems, so you break it down by field (maybe total is 99% but due_date is 80% because of date-format confusion) and by document type or vendor (one layout may be far worse). That breakdown tells you exactly where to invest — a better prompt for dates, an OCR cross-check for totals, a human step for the one bad layout.

Two subtleties make you sound experienced. First, define “correct” carefully: is 4210.0 equal to 4210.00? is a date correct in any format or only YYYY-MM-DD? Normalizing before comparison matters, exactly like comparing SQL result sets rather than strings (Track 9). Second, distinguish the cost of different errors: a missing field the system flags as low-confidence is cheap (a human fills it in), while a wrong field returned with confidence is expensive (it flows into a payment).

So you don’t just maximize accuracy — you tune the system to prefer “I’m not sure, please check” over confident wrong answers on the fields that matter. Designing extraction around calibrated uncertainty and human handoff, not just raw accuracy, is what turns a flashy demo into something a finance team will actually trust with their invoices.
