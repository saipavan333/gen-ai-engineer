Lab: Build a cost & latency calculator

Every serving decision is arithmetic you can do yourself. Build the model that turns tokens into dollars and seconds, then pull the four real levers — routing, caching, trimming, streaming — and watch a scary bill become a small one.

How to use this lab

Run the cells in order — they share memory. Prices are illustrative; the shape of the maths is what transfers.

Step 1 — Tokens to dollars

Cost is per token, and input and output are priced differently. Start with the estimator and the per-call formula.

step1_cost.py
def estimate_tokens(text):
    return max(1, len(text) // 4)          # ~4 characters per token in English

PRICES = {                                  # $ per 1M tokens (illustrative)
    "small": {"in": 0.15, "out": 0.60},
    "large": {"in": 3.00, "out": 15.00},
}

def cost_per_call(in_tokens, out_tokens, model="small"):
    p = PRICES[model]
    return (in_tokens * p["in"] + out_tokens * p["out"]) / 1000000

prompt = "You are a helpful support agent. " * 40
print("prompt tokens:", estimate_tokens(prompt))
print("small model per call: $", round(cost_per_call(800, 200, "small"), 6))
print("large model per call: $", round(cost_per_call(800, 200, "large"), 6))

Step 2 — Scale it to a real workload

Per-call costs look trivially small. Multiply by traffic and they stop being trivial. Also find out where the money actually goes.

step2_workload.py
IN_TOK, OUT_TOK, RPD = 800, 200, 20000      # tokens in, tokens out, requests/day

def monthly(in_t, out_t, rpd, model="small"):
    return cost_per_call(in_t, out_t, model) * rpd * 30

for m in ("small", "large"):
    print(m, "model -> $", round(monthly(IN_TOK, OUT_TOK, RPD, m), 2), "per month")

p = PRICES["large"]
in_share  = IN_TOK * p["in"]
out_share = OUT_TOK * p["out"]
total = in_share + out_share
print("\nof each call's cost:", round(in_share / total * 100), "% input,",
      round(out_share / total * 100), "% output")

Step 3 — Pull the levers

Now the part that pays your salary. Route easy requests to the cheap model, cache repeats, trim the prompt — and stack them.

step3_levers.py
def with_levers(in_t, out_t, rpd, cache_hit=0.0, cheap_share=0.0, trim=0.0):
    in_t = in_t * (1 - trim)              # shorter prompt
    paid = rpd * (1 - cache_hit)          # cached calls cost nothing
    cheap = paid * cheap_share
    big   = paid - cheap
    daily = (cheap * cost_per_call(in_t, out_t, "small")
             + big   * cost_per_call(in_t, out_t, "large"))
    return daily * 30

print("everything on the large model : $", round(with_levers(IN_TOK, OUT_TOK, RPD), 2))
print("+ route 80% to the small model: $", round(with_levers(IN_TOK, OUT_TOK, RPD, cheap_share=0.8), 2))
print("+ 40% cache hit rate          : $", round(with_levers(IN_TOK, OUT_TOK, RPD, 0.4, 0.8), 2))
print("+ trim the prompt by 30%      : $", round(with_levers(IN_TOK, OUT_TOK, RPD, 0.4, 0.8, 0.3), 2))

Step 4 — Latency, and what streaming really does

Latency splits in two: prefill (reading your prompt → time to first token) and decode (writing the answer, one token at a time). Long prompts delay the start; long answers stretch the finish.

step4_latency.py
def latency(in_tokens, out_tokens, prefill_per_1k=0.12, per_out_token=0.012):
    ttft   = 0.25 + (in_tokens / 1000) * prefill_per_1k   # time to first token
    decode = out_tokens * per_out_token
    return ttft, ttft + decode

for label, i, o in [("short prompt", 200, 200),
                    ("long prompt ", 4000, 200),
                    ("long answer ", 800, 800)]:
    ttft, total = latency(i, o)
    print(label, "| first token", round(ttft, 2), "s | full answer", round(total, 2), "s")

print("\nStreaming doesn't change the total — it changes when the user sees word one.")
print("That's why a long prompt hurts perceived speed more than a long answer.")
What you built
  • A cost model: tokens → dollars, input vs output priced separately.
  • A workload projection that turns fractions of a cent into a monthly bill.
  • A lever simulator: routing, caching, trimming — stacked.
  • A latency model separating time-to-first-token from decode time.

You can now answer “what will this cost and how fast will it feel?” with arithmetic instead of a shrug — which is exactly what gets asked in system-design rounds.

Make it yours

1. Your feature has a 5,000-token prompt and a 100-token answer. Which lever helps most — trimming the prompt or shortening the answer? Prove it with the code.
2. An agent makes 6 calls per task. Change the model to charge per task, not per call. What happens to the monthly figure?

▸ Show reasoning

1. Trimming the prompt, by a wide margin — with 5,000 in and 100 out, input dominates both the token count and the time-to-first-token, even though output is priced higher per token. Always optimise where the tokens actually are, which is why you measure before you tune.

2. Roughly 6× — and worse in practice, because each step re-sends a growing history, so input tokens balloon across the task. This is why agents need per-task budgets, not per-call ones; a "cheap" model in a chatty loop is not cheap.
