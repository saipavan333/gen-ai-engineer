Watch tokens become time and money

Every architecture argument about cost and latency comes down to one asymmetry: input is read all at once, output is written one token at a time. Watch a single request move through both, and see where the seconds and the cents actually go.

How to use this lab

Press ▶ Play to follow one request from prompt to reply, or Step through each stage. Drag the input and output token sliders and watch the cost and latency respond.

Make it yours

1. Set output tokens to their maximum and input to its minimum. Which dominates the cost now — and why, when there are far fewer output tokens than input?
2. Streaming doesn't change the total time at all. So why does it make the product feel dramatically faster?

▸ Show reasoning

Output dominates because each output token costs several times an input token and is generated serially — one forward pass per token — while the entire input is processed in a single parallel prefill pass. So a short reply can still cost and take more than a long prompt. That's why the biggest, cheapest lever is usually capping reply length, not shortening the prompt. Streaming feels faster because perceived latency is dominated by time-to-first-token: the user sees words appear almost immediately and starts reading, even though the full response finishes at the same moment. Speed the user feels is a product decision layered on top of the engineering one.
