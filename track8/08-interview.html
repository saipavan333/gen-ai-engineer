Interview check: Serving, Latency & Cost

Serving questions test whether you can take a working LLM feature and make it affordable and fast at scale. Interviewers want a clear cost/latency model and the right lever for each bottleneck. Here’s the set.

The drill

Answer aloud first. The throughline: cost and latency are driven by tokens; to go cheaper cache → route → smaller models → trim tokens (→ batch if self-hosting); to go faster stream → shrink prompt → shorter output → faster model; and always measure first.

The questions teams ask

Q1 · The model

“What drives the cost and latency of an LLM call?”

▸ Strong answer

Tokens. Cost ≈ (input + output tokens) × price, with output usually priced higher. Latency ≈ time-to-first-token (prefill — the model reading the whole prompt) plus output tokens × time-per-token (decode). So a long prompt mainly raises TTFT and input cost; a long answer raises decode latency and output cost. Output tokens are expensive on both axes, and for agents/RAG I estimate cost per task (all the calls), not per call.

Q2 · Latency UX

“The chat feels slow but the API speed is fixed. What do you do?”

▸ Strong answer

Stream the output — show tokens as they’re generated so the first words appear in a fraction of a second. Total time is unchanged but perceived latency drops dramatically, which is what users react to. I’d also shrink the prompt (lowers TTFT) and cap/shorten the output, and consider a faster model for the easy path. Streaming is the default for any chat UI — except where the consumer is code (parsing JSON/tool calls), where I collect-then-parse.

Q3 · Cost

“Costs are too high at scale. Where do you start?”

▸ Strong answer

Measure where the tokens and calls go (Track 6 traces), then apply levers in order of ROI: caching (the biggest single win — repeats are instant and near-free; exact + semantic + prefix), model routing (send the easy majority to a cheap model, reserve the big one for hard cases), smaller/quantized models, and token budgeting (trim the largest prompt component). If self-hosting, add continuous batching to keep GPUs busy. The key is to find the actual bottleneck first — don’t optimize the part that isn’t the problem.

Q4 · Routing

“How does model routing save money without tanking quality?”

▸ Strong answer

Most traffic is easy and doesn’t need a frontier model, so a router (rules, a small classifier, or a cascade that tries cheap-then-escalates) sends easy queries to a cheap, fast model and hard ones to a capable model. For easy queries a small model gets ~99% of the quality at a fraction of the cost, so you save a lot while the hard minority still gets full capability. I validate small-path quality on an eval set and monitor the escalation rate, tuning the router with data.

Q5 · Self-hosting

“When does self-hosting make sense, and what are the key levers?”

▸ Strong answer

Self-host at scale (no per-token fees, just hardware), for privacy (data never leaves — Track 7), or for custom fine-tunes. The key serving levers are quantization (fewer bits per weight → fits cheaper GPUs, faster, small quality drop — measure it) and continuous batching via a serving engine like vLLM (process many requests per GPU → high throughput → low cost per request). With hosted APIs the provider already does these, so my levers there are routing, caching, and token budgeting.

Q6 · Caching

“What are the risks of caching LLM responses?”

▸ Strong answer

Serving stale or wrong answers. Don’t cache personalized responses (one user could get another’s), time-sensitive answers (today’s price), or intentionally-varied creative output. For a semantic cache, a too-loose similarity threshold returns a near-miss answer to a different question — tune it on real query pairs. And define invalidation: tie cache expiry to source-data changes (or use a TTL) so updated docs don’t serve old answers. Cache the stable, shared, repeatable stuff with sane invalidation.

Rapid-fire self-check

A response feels sluggish: nothing appears for ~3 seconds, then text streams quickly. The prompt is 6,000 tokens. The biggest fix for that initial delay is…

  • A) Shrink the prompt — the long prompt makes prefill slow, which is what delays the first token (TTFT).
    Correct. A 3-second gap before any text, then fast streaming, is the signature of high TTFT from a long prompt (prefill must read all 6,000 tokens first). Reducing the input context directly lowers TTFT — and usually cost and quality too.
  • B) Generate more output tokens.
    More output adds decode time, not less initial delay. The gap is before generation starts (prefill), which output length doesn’t affect.
  • C) Disable streaming.
    That would make it feel worse — you’d wait for the whole answer. The initial delay is prefill; streaming is already helping once tokens start.
  • D) Raise the temperature.
    Temperature doesn’t affect TTFT or prompt-processing time. The fix is a shorter prompt (or faster model).

Which single lever is most often the biggest cost win for a high-traffic LLM app?

  • A) Caching — real traffic is repetitive, and cache hits are instant and nearly free.
    Correct. Production traffic has a fat head of common queries, so caching turns a large share of calls into near-free lookups — frequently the single biggest cost reducer, and it lowers latency too. (Routing and token budgeting are close behind.)
  • B) Raising the temperature.
    Temperature has no effect on cost. It’s unrelated to tokens, calls, or hardware.
  • C) Using a bigger model everywhere.
    That increases cost across all traffic — the opposite of a saving. Right-sizing (routing) and skipping calls (caching) are the wins.
  • D) Longer outputs.
    Longer outputs cost more on both axes. The biggest cost wins come from skipping or right-sizing calls and trimming tokens, not adding output.

You add streaming and the total response time is unchanged, yet users say it feels much faster. Why?

  • Time-to-first-token drops — perceived latency is driven by when words start appearing.
    Correct. Users start reading immediately, so the experience feels fast even though the full response finishes at the same moment.
  • Streaming reduces the total tokens generated.
    It doesn’t change how many tokens are produced.
  • Streaming lowers the price per token.
    Cost per token is unchanged by streaming.
  • The model thinks less.
    The same computation happens; only delivery changes.

Output tokens cost several times more than input and are generated one at a time. What is usually the biggest lever on cost and latency?

  • Cap or shorten the output length.
    Correct. Output dominates both bill and latency because it’s pricier per token and serial, so limiting reply length is the highest-leverage move.
  • Shorten the input prompt.
    Input is cheaper and processed in parallel; trimming it helps less.
  • Raise the temperature.
    Unrelated to cost or latency.
  • Add more few-shot examples.
    That increases input tokens, the opposite of helping.
Can you explain each of these?
  • The token-driven cost & latency model (prefill/TTFT vs decode; input vs output).
  • Streaming and perceived latency (and when not to stream).
  • Caching (exact / semantic / prefix) and its correctness risks.
  • Model routing and cascades; the cost/quality frontier.
  • Quantization (bits → memory/speed/quality) and when it applies.
  • Batching/throughput and the latency trade-off (self-hosting).
  • Token budgeting: measure where tokens go, trim the biggest.
