Lab: Build a mini language model

You’ve learned how the engine works. Now build one. In about forty lines of plain Python you’ll write a tokenizer, a tiny next-token model, and a temperature sampler — then watch it generate text. Everything here runs right in this page.

Step 1 — Tokenize

A model never sees letters, it sees tokens. Write a simple splitter that keeps punctuation separate and lets a space ride along with the word after it — the same behaviour real sub-word tokenizers have.

step1_tokenize.py
import re

def tokenize(text):
    # words, numbers and punctuation — a stand-in for real sub-word tokenizing
    return re.findall(r"\s?[A-Za-z]+|\s?[0-9]+|\s?[^\sA-Za-z0-9]+", text)

text = "Retrieval grounds the model in your documents."
toks = tokenize(text)
print(toks)
print("tokens:", len(toks), "| characters:", len(text))

Step 2 — Learn what follows what

A language model is, at heart, a table of “given this, what usually comes next?”. Count every pair of neighbouring words in a small corpus. That count table is your model.

step2_model.py
corpus = """the model predicts the next token
the model reads the prompt carefully
the next token depends on the prompt
a model can predict the next word"""

words = corpus.split()
bigrams = {}
for a, b in zip(words, words[1:]):
    bigrams.setdefault(a, {})
    bigrams[a][b] = bigrams[a].get(b, 0) + 1

print("after 'the'   ->", bigrams["the"])
print("after 'model' ->", bigrams["model"])

Step 3 — Sample with a temperature

Counts become probabilities through a softmax, and temperature reshapes that distribution before you draw from it. Low temperature sharpens toward the favourite; high temperature flattens everything toward a coin-flip. Run this and watch the numbers move.

step3_sample.py
import math, random

def sample(counts, temperature=1.0):
    options = list(counts.keys())
    logits  = [math.log(counts[w]) for w in options]
    scaled  = [l / max(temperature, 0.01) for l in logits]
    top     = max(scaled)
    exp     = [math.exp(s - top) for s in scaled]
    total   = sum(exp)
    probs   = [e / total for e in exp]
    r, acc  = random.random(), 0.0
    for w, p in zip(options, probs):
        acc += p
        if r <= acc:
            return w, {o: round(p, 2) for o, p in zip(options, probs)}
    return options[-1], {}

counts = {"next": 3, "prompt": 1, "model": 1}
for t in [0.2, 1.0, 3.0]:
    pick, dist = sample(counts, t)
    print("temperature", t, "-> picked", pick, "  distribution", dist)
What just happened

At 0.2 the distribution is almost all on “next” — nearly deterministic. At 3.0 the three options approach equal probability — near-random. That single number is the entire creativity dial you tune in production, and you just implemented it.

Step 4 — Generate (autoregression)

Now the loop that makes it a generator: pick a token, append it, feed the result back in, repeat. That’s autoregression — the same loop running inside every model you’ll ever call.

step4_generate.py
def generate(start, n=8, temperature=0.8):
    out, word = [start], start
    for _ in range(n):
        if word not in bigrams:
            break
        word, _ = sample(bigrams[word], temperature)
        out.append(word)
    return " ".join(out)

for t in [0.3, 1.5]:
    print("temp", t, "->", generate("the", 8, t))
Make it yours

1. Paste your own paragraph into corpus in Step 2, re-run Steps 2 and 4, and see it speak in your text’s voice.
2. In Step 4, set temperature=0.05. Why does it repeat itself?
3. Change tokenize to split on characters instead of words. What happens to the output quality — and why does token choice matter so much?

▸ Show reasoning

2. At near-zero temperature the softmax collapses onto the single highest-count option every time, so the model becomes deterministic and falls into a loop (“the next token the next token…”). This is exactly why production systems rarely use temperature 0 for open-ended writing — and always use it for extraction, where you want the same answer twice.

3. Character tokens make the sequences far longer and the statistics far weaker per step, so output degrades into noise. Real tokenizers sit in the sweet spot — sub-word pieces that are frequent enough to have good statistics but small enough to cover any word. You just felt why tokenization is a design decision, not a detail.

What you built
  • A tokenizer that turns text into the units a model actually sees.
  • A next-token model — a probability table learned from data.
  • A softmax + temperature sampler, the real creativity dial.
  • An autoregressive loop — the mechanism behind every LLM response.

A frontier model differs in scale and in learning richer patterns than word pairs — but the shape you just built is the shape of the real thing.
